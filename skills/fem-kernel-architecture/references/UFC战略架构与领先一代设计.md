# UFC 战略架构与「领先一代」设计

> 以 UFC 软件架构为基础，以 ABAQUS 等成熟主流商业软件为锚点，借鉴并超越，达到**领先一代**水准。
> 本文档拔高思维高度，做全面审视与意见梳理，并衔接架构设计、算法框架更新与技能包完善。
> 可与 `/bmad-brainstorming`、`/bmad-party-mode`、`/bmad-help`、`/bmad-review-adversarial-general` 等配合使用。

---

## 一、「领先一代」的具象定义

不是简单功能对齐，而是在**架构、算法、工程、生态**四个维度上形成可衡量的代差。

| 维度           | ABAQUS/主流现状（锚点）                                 | 「领先一代」目标（UFC）                                                                                                                                                            |
| -------------- | ------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **架构** | 单体/模块边界模糊、历史包袱、跨层隐式依赖               | 六层+域级+四型+四链**全链路可追溯**；Bridge 显式、无循环依赖；模块可替换、可独立测试                                                                                         |
| **算法** | 隐式/显式分立、多物理场后补、时间积分与接触策略分散     | **统一场论**（C·φ̇+K·φ+f_int=f_ext）+ **统一时间积分**（θ/Newmark/广义α）+ **可插拔耦合策略**（单块/交替/分区）；接触/单元/本构在同一数据与调用契约下扩展 |
| **工程** | 验证用例分散、回归不系统、文档与代码脱节、并行/GPU 后补 | **验证驱动**：每算法/本构/单元族对应标准题+CI 回归；**数据契约+可观测**统一；**并行/异构从设计阶段**纳入（Workspace/Backend 抽象）                               |
| **生态** | 脚本与二次开发受限、API 封闭、绑定单一                  | **开放扩展点**（UEL/UMAT/载荷注册）+ **稳定 API**+**多语言绑定**（Python/C）；模块化发布、API 版本承诺                                                           |

**一句话**：领先一代 = **架构可演进、算法可统一、工程可重复、生态可扩展**，且四者在同一套契约与技能包下闭环。

## 二、以 ABAQUS 为锚：借鉴什么、超越什么

### 2.1 值得借鉴（保留并规范化）

| 方面                     | ABAQUS 长处                                    | UFC 中的对应与强化                                                                                   |
| ------------------------ | ---------------------------------------------- | ---------------------------------------------------------------------------------------------------- |
| **关键字与模型树** | INP/CAE 模型树清晰、Step/Load/BC/Material 分层 | L3_MD 域级与 00-域级划分规范 已覆盖；强化**KW→Desc 的单一映射** 与校验                        |
| **单元与本构体系** | 单元族+本构库+截面+材料 组合完整               | L4_PH 单元/材料/接触 与 L3_MD 的 Desc 对齐；**TL/UL 与单元族范式**（99 文档）统一扩展          |
| **求解器组合**     | 隐式/显式、线性/非线性、直接/迭代、特征值      | L2_NM 作为唯一数值层入口；**求解器选择器（Solver_Optimizer）** 与 Bridge 模式（02-04）统一封装 |
| **验证与基准**     | 验证手册、NAFEMS 等基准题                      | 02-07 验证体系 +**fem-kernel-verification**：标准题清单、CI 回归、与 ABAQUS 结果对照           |
| **用户子程序**     | UEL/UMAT/UVARM 等扩展入口                      | **fem-kernel-extensibility**：注册与 Bridge、契约化接口、不泄漏内核类型                        |

### 2.2 要超越（代差所在）

| 方面                   | ABAQUS/主流短板                 | UFC 超越路径                                                                                                                                           |
| ---------------------- | ------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **架构透明度**   | 黑盒、依赖与数据流难追溯        | **四链（理论/逻辑/计算/数据）** 显式化；每层每域 **Desc/Algo/Ctx/State** 文档化；变更可做**影响分析**（fem-kernel-doc-traceability） |
| **算法统一性**   | 隐式/显式两套路径、多物理场拼凑 | **统一场论**（04-01/04-02 中的 UF_UnifiedField_Core）+ **统一时间积分框架**；多物理场为**同一方程+不同策略**（单块/交替/分区）       |
| **质量与回归**   | 验证分散、回归不自动化          | **验证驱动**：新算法/本构/单元必带标准题+回归；**fem-kernel-test-design** 保证可测性与 Mock 边界；CI 与 02-07 绑定                         |
| **扩展与 API**   | 子程序接口固定、难以扩展新物理  | **扩展点+稳定 API**：注册制 UEL/UMAT/载荷；**fem-kernel-api-design** 划定稳定边界与兼容策略；Python/C 绑定基于契约                         |
| **并行与异构**   | 多核/GPU 多为后补               | **Workspace/Backend 抽象** 从设计阶段纳入 L2_NM/L5_RT；**fem-kernel-parallel-design**（待建）指导统一并行抽象                              |
| **可观测与运维** | 中间状态与性能难观测            | **fem-kernel-observability**：错误与 02-06 一致、性能计数与 L1_IF Timer 统一、诊断可关闭                                                         |

## 三、架构设计：在现有六层上做「拔高」

### 3.1 坚持与强化（已有基础）

- **六层+严格依赖方向**：L1←L2←L3←L4←L5←L6，禁止反向；跨层仅通过 **Bridge**（02-04）。
- **域级+四型**：每域 Desc/Algo/Ctx/State 明确；**接口仅传结构体名**（UFC_数据结构_分层分域分模块结构体规划）。
- **Context 传递**：显式 Ctx 替代线程局部；错误经 Ctx 传播（02-06），与 **fem-kernel-observability** 一致。
- **四链**：理论链、逻辑链、计算链、数据链 作为**评审与变更分析**的固定视角（fem-kernel-architecture）。

### 3.2 需要补强与显式化的设计

| 设计项                  | 说明                                                                                                                | 与技能/文档的衔接                                    |
| ----------------------- | ------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------- |
| **统一场论入口**  | UF_UnifiedField_Core：场类型、场状态、统一方程 C·φ̇+K·φ+f_int=f_ext；所有物理（结构/热/流/耦合）在此框架下表达 | 04-02 阶段 C；算法框架见下节                         |
| **统一时间积分**  | θ/Newmark/广义α 作为可插拔策略；Step 驱动调用统一接口，而非隐式/显式两套分支                                      | 04-02 C.2；L2_NM TimeInt 与 L5_RT Step 的契约        |
| **多物理场耦合**  | 单块（ monolithic）/交替（staggered）/分区迭代 三种策略，接口统一、可替换                                           | 04-02 C.3；L2_NM Ctx 中的耦合模块                    |
| **扩展点契约**    | UEL/UMAT/用户载荷 的输入输出类型、注册 API、错误返回 写入**数据契约** 与 02-04 Bridge 文档                    | fem-kernel-extensibility + fem-kernel-data-contract  |
| **并行/异构抽象** | Workspace 按线程/进程/设备划分；Backend（CPU/GPU）在 L2_NM 与 L5_RT 装配层可插拔                                    | fem-kernel-parallel-design（待建）；04-01 求解层差距 |

### 3.3 架构评审的固定问题（拔高用）

每次重大变更或阶段评审时，用下列问题自检：

1. **四链**：该变更是否破坏了理论链/逻辑链/计算链/数据链中的任一条？是否更新了对应文档？
2. **依赖**：是否仍保持严格单向、无循环？新增 Bridge 是否在 02-04 中显式列出？
3. **契约**：跨层/跨域类型是否在 **fem-kernel-data-contract** 的契约表中有条目？版本与兼容是否明确？
4. **验证与测试**：新算法/本构/单元是否在 02-07 与 **fem-kernel-verification** 下有标准题或回归用例？可测性是否满足 **fem-kernel-test-design**？
5. **扩展与 API**：若涉及对外接口，是否符 **fem-kernel-api-design** 的稳定边界与兼容策略？

## 四、算法框架更新：统一场论与可插拔策略

### 4.1 统一场方程（核心抽象）

- **目标**：所有物理（结构、热、流、电磁、耦合）统一为**C·φ̇ + K·φ + f_int = f_ext**其中 φ 为场自由度，C 为阻尼/质量类，K 为刚度/传导类，f_int/f_ext 为内力/外力。
- **落地**：UF_UnifiedField_Core（04-02 阶段 C.1）；L3_MD 的场定义 → L4_PH 的本构与单元贡献 → L5_RT 组装 → L2_NM 求解；**数据链**上类型一致（Desc/Ctx/State）。
- **与 ABAQUS 的差异**：ABAQUS 隐式/显式两套路径；UFC 用**同一方程+不同时间积分策略**（θ/Newmark/广义α）覆盖，显式视为积分策略的一种。

### 4.2 统一时间积分框架

- **策略可插拔**：θ 法、Newmark、广义 α 等实现为同一接口的不同策略；Step 类型（Static/Dynamic/Heat 等）仅选择策略与参数，不复制整条求解路径。
- **与 L2_NM TimeInt 的契约**：L5_RT 传入当前状态与步长，L2_NM 返回更新后状态与可选刚度/阻尼贡献；接口用 **State/Ctx** 传递，见 **fem-kernel-data-contract**。
- **文档**：在 02-05 数据流与 L2_NM 设计文档中明确「时间积分作为统一场方程的时间离散化」，与 01-理论基础 对应。

### 4.3 多物理场耦合策略

- **单块**：所有场自由度在同一系统矩阵中求解；适合强耦合、小规模。
- **交替**：场间按序更新，如先结构后热；适合弱耦合或已有独立求解器。
- **分区迭代**：子域或子场分别求解，通过界面条件迭代；适合大规模、可并行。
- **实现**：L2_NM Ctx 中多物理场耦合模块（04-02 C.3）；策略可插拔，与 Bridge 模式一致。

### 4.4 接触、单元、本构的扩展一致性

- **接触**：接触对、摩擦、自适应 在 L4_PH 与 L5_RT 的契约下扩展；与统一场方程中的 f_int/f_ext 贡献一致。
- **单元**：单元族（壳/梁/实体/耦合/特殊）通过 **TL/UL 范式**（99 文档）与 L3_MD ElementType/ElemFormulation 对齐；新增单元族按同一模板扩展。
- **本构**：材料库与 UMAT 通过 **fem-kernel-extensibility** 的注册与 Bridge 接入；本构接口（应变→应力、切线）与 02-07 验证（客观性、热力学一致性）对齐。

## 五、技能包完善：闭环与优先级

### 5.1 已有技能与角色

| 技能                               | 角色                                                                  |
| ---------------------------------- | --------------------------------------------------------------------- |
| **fem-kernel-architecture**  | 总纲：六层/域级/四型/四链、域级四阶段、依赖与 Bridge 检查、测试与验收 |
| **fem-kernel-verification**  | 验证体系落地、标准题、CI 回归、与 02-07 对齐                          |
| **fem-kernel-test-design**   | 可测性、单元/集成/端到端、Mock/注入                                   |
| **fem-kernel-data-contract** | 层间/域间契约、版本兼容、数据链显式化                                 |
| **fem-kernel-observability** | 性能/诊断/日志、与 02-06 错误架构对齐                                 |
| **fem-kernel-extensibility** | UEL/UMAT/载荷 扩展点、注册与 Bridge                                   |
| **fem-kernel-api-design**    | 对外 API 边界、兼容与废弃、语言绑定                                   |

### 5.2 建议补全的技能

| 技能                                       | 用途                                                                               | 优先级                |
| ------------------------------------------ | ---------------------------------------------------------------------------------- | --------------------- |
| **fem-kernel-parallel-design**       | Workspace/Backend 抽象、并行与 GPU 在 L2_NM/L5_RT 的对接、从设计阶段避免后补       | 高（与阶段 D/E 并行） |
| **fem-kernel-doc-traceability**      | 四链文档化、理论→实现、需求→用例、变更→影响分析                                 | 中（与文档体系同步）  |
| **fem-kernel-unified-field**（可选） | 统一场论与统一时间积分的**设计检查清单**与实现模板；与 04-02 阶段 C 直接对应 | 中（算法框架落地时)   |

### 5.3 技能包与「领先一代」的对应

- **架构可演进** → fem-kernel-architecture + fem-kernel-data-contract + fem-kernel-doc-traceability
- **算法可统一** → 统一场论/时间积分/耦合（项目文档 04-02）+ 可选 fem-kernel-unified-field
- **工程可重复** → fem-kernel-verification + fem-kernel-test-design + fem-kernel-observability
- **生态可扩展** → fem-kernel-extensibility + fem-kernel-api-design

### 5.4 技能包使用顺序（建议）

1. **做架构/域级决策时**：fem-kernel-architecture 为主，辅以 fem-kernel-data-contract、依赖与 Bridge 检查。
2. **做算法/统一场/时间积分时**：查 04-02 与 02-05；必要时建 fem-kernel-unified-field 或在其文档中写检查清单。
3. **做实现与交付时**：fem-kernel-verification + fem-kernel-test-design 保证质量；fem-kernel-observability 保证可观测。
4. **做扩展与对外 API 时**：fem-kernel-extensibility + fem-kernel-api-design；契约更新走 fem-kernel-data-contract。
5. **做并行/GPU 时**：fem-kernel-parallel-design（待建）；与 L2_NM/L5_RT 设计同步。
6. **做变更影响分析或文档梳理时**：fem-kernel-doc-traceability（待建）+ 四链审视。

## 六、对抗性审视与风险

### 6.1 可能被质疑的点

| 质疑                                   | 回应与缓解                                                                                                 |
| -------------------------------------- | ---------------------------------------------------------------------------------------------------------- |
| 「统一场论」是否过度抽象、拖慢交付？   | 先在小范围落地（如结构+热），再推广；04-02 阶段 C 分步实施；不要求一步到位。                               |
| 技能包过多，是否流于形式？             | 技能是**检查清单与文档锚点**，不替代实现；每次迭代只选用 1～2 个技能深入，其余作参考。               |
| 超越 ABAQUS 的「领先一代」是否不现实？ | 定义在**架构/工程/生态**上可衡量、可逐步达成；**算法**上先统一再扩展，不追求功能数量一步对标。 |
| 并行/GPU 从设计阶段纳入是否过早？      | Workspace/Backend 抽象可先做接口与占位实现，具体后端可后续替换；避免日后大改。                             |

### 6.2 风险与依赖

- **人力与节奏**：04-02 路线图阶段多、周期长；建议按阶段设置**可验证里程碑**（如依赖审计通过、首个统一场试点、首条 CI 回归链）。
- **文档与代码同步**：四链与契约表易滞后；将「更新契约/四链」纳入**完成定义**（如 PR 或阶段验收条件）。
- **ABAQUS 对照**：验证与回归需与 ABAQUS 结果对照时，需稳定环境与用例；**fem-kernel-verification** 中明确「对照用例集」与容差策略。
