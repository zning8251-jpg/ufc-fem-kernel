# 有限元内核可观测性技能

本技能用于在六层架构下设计**可观测性**：性能计数、计时钩子、诊断接口、日志与错误传播的一致性，以及可选的追踪与调试支持，便于运维、调优与排错。

---

## 何时使用本技能

- 用户提到 **可观测**、**性能分析**、**诊断**、**计时**、**日志**、**追踪**、**调试**、**因果**、**为何变更**
- 需要为**某域或某层**增加性能计数、计时点或诊断输出
- 讨论 **错误传播**、**错误码体系**、**与 L1_IF 错误/日志的对接**
- 设计 **Checkpoint/重启** 时的状态导出、或 **Profiling** 所需的钩子

## 可观测维度


| 维度         | 内容                                                                        |
| ---------- | ------------------------------------------------------------------------- |
| **错误与状态**  | 与 02-06 一致：错误码范围按层划分、通过 Ctx 传播、可恢复/致命/警告策略；日志使用 L1_IF Log 或等价接口           |
| **性能计数**   | 关键路径的计时（如 Step/Increment、组装、求解、本构）；可选：FLOP、内存带宽估算；输出与 L1_IF Timer 或等价模块对齐 |
| **诊断接口**   | 按需导出：当前步/增量、收敛残差、选定的 Ctx/State 快照；用于调试与离线分析，不改变主路径逻辑                      |
| **追踪（可选）** | 调用链或关键事件序列；轻量实现（如层级/域标识 + 时间戳），避免主路径膨胀                                    |

## 因果叙事（横切视图，非第五链）

**定位**：回答「为何发生 / 由谁触发 / 上游是什么」的叙事需求**不**引入与理论/逻辑/计算/数据并列的**第五条链**；不新增合同枚举（如域 `CONTRACT.md` 仍只要求四链各一句可核对说明）。因果信息挂在**可观测性**上，作为 trace/span 或诊断记录的**可选标注维度**。

**推荐标注字段（实现时按需取舍，保持轻量）**：


| 标注                   | 含义                                           |
| -------------------- | -------------------------------------------- |
| `trigger` / `reason` | 触发原因：例如步进阶段、迭代分支、Populate 阶段、Harness 钩子名     |
| `upstream`           | 上游标识：层/域/过程名或稳定合同字段名（不写整结构 dump）             |
| `chain_hint`         | 四链归类提示：本事件主要服务于理论/逻辑/计算/数据中的哪一条（可多选，仍以四链为闭集） |


**与四链的分工**：**理论链**给出方程与不变量依据；**逻辑链**给出调度与调用次序；**计算链**给出数值算子与复杂度；**数据链**给出真相源与读写方向。**因果视图**把上述信息压缩为排障可读的一句话或一对 `(trigger, upstream)`，便于 Span 关联与回归对比，而不是替代四链分类。

**文档侧写法**：在域说明或设计笔记中可增加简短 **「因果说明」** 小节，用两三句话说明「本域状态变更由哪些上游域/阶段触发、四链中哪几条共同解释 why」，**不**扩展「五链」用词。

## 可观测性检查清单

- 新增错误是否使用 02-06 的**错误码范围**并写入 Ctx 的 err_status？
- 错误是否沿 **Context 向上传播**，不跨层直接访问？
- 计时/计数是否通过 **L1_IF Timer** 或统一接口，便于关闭或替换实现？
- 诊断输出是否**可关闭或按级别过滤**，避免生产环境噪音？
- 观测点是否**不破坏层间依赖**（不因观测而增加反向依赖）？
- 若记录「因果」类信息，是否仅作为 **trace/诊断元数据**，且 **chain_hint 仍落在四链闭集**内？

---

## 产出建议

- **观测点表**：按层/域列出计时点、计数项、诊断接口、对应实现位置（如 IF_Err_*.f90、RT_Err_*.f90）。
- **与 02-06 的同步**：新增错误码时更新 02-06 表格；新增恢复策略时在文档中说明。
- **日志/输出约定**：日志级别、模块名、与 L1_IF Log 的对接方式。

## 与 fem-kernel-architecture 的衔接

- 架构技能中的依赖与 Bridge 检查保证观测不引入反向依赖；错误传播方式与 02-03 Context、02-06 一致。
- 域级迭代「阶段 2：设计决策」中若涉及错误码或性能钩子，应在 02-06 与本技能下统一约定，并在「阶段 4：文档沉淀」中更新观测点表。
