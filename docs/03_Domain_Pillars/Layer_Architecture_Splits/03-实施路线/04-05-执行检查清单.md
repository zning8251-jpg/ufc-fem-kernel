## 合并前检查

- [ ] **文件分析**：统计待合并文件的行数、子程序数量
- [ ] **子程序分析**：识别每个子程序的行数，标记 > 3000 行的子程序
- [ ] **依赖分析**：分析合并对依赖关系的影响
- [ ] **功能分析**：确认合并的模块功能相关
- [ ] **测试分析**：确认合并后测试仍能覆盖

## 合并执行

- [ ] **备份**：合并前创建 Git 分支
- [ ] **合并实现**：将代码合并到目标模块
- [ ] **更新 USE**：修改所有引用旧模块的 USE 语句
- [ ] **CMakeLists**：更新 CMakeLists.txt，排除旧文件
- [ ] **子程序拆分**：如果合并后子程序 > 4000 行，立即拆分

## 合并后验证

- [ ] **编译**：确保代码能编译通过
- [ ] **子程序长度**：验证所有子程序 ≤ 4000 行
- [ ] **文件长度**：验证所有文件 ≤ 3000 行
- [ ] **功能测试**：运行相关测试用例
- [ ] **依赖检查**：运行依赖审计工具验证
- [ ] **文档更新**：更新相关文档

## 质量指标

| 指标 | 目标值 | 检查方法 |
| -------------- | ---------------- | ---------------- |
| 文件平均行数 | 1000-2000 行 | `wc -l *.f90` |
| 子程序平均行数 | 300-800 行 | 代码分析工具 |
| 最长子程序 | ≤ 4000 行 | 代码分析工具 |
| 最长文件 | ≤ 3000 行 | `wc -l *.f90` |
| 模块数量 | ≤ 50 个核心模块 | 文件统计 |
| 循环依赖 | 0 个 | `dep_audit.py` |
| 测试覆盖率 | ≥ 80% | 测试工具 |
