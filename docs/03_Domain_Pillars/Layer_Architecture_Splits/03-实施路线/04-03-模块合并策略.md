## 核心原则

**双重约束**：既要控制模块数量，又要保证代码可维护性

| 约束维度 | 限制 | 理由 |
| -------------------- | ---------------------- | --------------------------------- |
| **文件大小** | ≤ 3000 行 | 便于 IDE 加载、Git 合并、代码审查 |
| **子程序大小** | ≤ 4000 行（硬性上限） | 便于理解、调试、测试 |
| **理想子程序** | 200-1000 行 | 单一职责、易于维护 |
| **模块数量** | 核心域 ≤ 50 个 | 降低依赖复杂度 |

## 合并策略

### 何时合并

1. **功能相关**：模块功能高度相关，经常一起使用
2. **依赖紧密**：模块间依赖关系紧密，难以独立使用
3. **文件较小**：合并后文件大小 ≤ 3000 行
4. **减少依赖**：合并后能减少模块间依赖关系

### 合并方法

1. **合并实现**：将独立模型/子模块的完整实现合并进聚合模块
2. **更新 USE**：修改 Interface、Registration、Reg 等对旧模块的 `USE` 语句
3. **CMakeLists 排除**：在 `ufc_core/CMakeLists.txt` 中添加 `list(FILTER UFC_CORE_SRCS EXCLUDE REGEX ...)` 排除已合并的旧文件
4. **删除旧文件**：删除已被合并的独立源文件

## 子程序拆分策略

### 何时拆分

1. **子程序过大**：子程序行数 > 4000 行（硬性上限）
2. **职责不清**：子程序承担多个职责
3. **难以测试**：子程序难以进行单元测试
4. **难以维护**：子程序逻辑复杂，难以理解和修改

### 拆分方法

1. **按功能拆分**：将子程序按功能拆分为多个子程序
2. **提取公共逻辑**：提取公共逻辑为独立子程序
3. **使用辅助子程序**：将复杂逻辑提取为辅助子程序
4. **保持接口稳定**：拆分后保持公共接口不变
