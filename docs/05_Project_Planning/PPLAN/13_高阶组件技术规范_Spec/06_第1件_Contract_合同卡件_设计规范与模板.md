# 第 1 件：Contract (合同卡件) 设计规范与模板

> **文档位置**：`UFC/docs/05_Project_Planning/PPLAN/13_高阶组件技术规范_Spec/06_第1件_Contract_合同卡件_设计规范与模板.md`  
> **所属套件**：全景套件 v3.0 第 1 件  
> **目标**：以合同驱动开发（Contract-Driven Development），锁定每个域的职责边界、跨层依赖、四型映射与 SIO 签名。

---

## 1. 核心目标与技术红线

### 1.1 核心目标
1. **边界仲裁**：当开发人员对于“某段代码该放在哪个文件”产生争议时，`CONTRACT.md` 是最终仲裁依据。
2. **防腐契约**：明确列出该域的输入、输出、依赖。防止出现跨层直接引用的“面条代码”。

### 1.2 架构红线 (Red Lines)
- **无合同不编码**：任何新建的物理域或子系统，必须先提交 `CONTRACT.md` 审核。
- **所见即所得**：代码的 `USE` 依赖、子程序签名必须与 `CONTRACT.md` 完全一致。如果实现有变，必须首先修改合同。

---

## 2. 合同模板与必填项

每个子域的根目录下必须存在一个 `CONTRACT.md`，其标准模板如下：

```markdown
# <层级>/<域> (<Domain_Name>) 合同卡

## 1. 域职责定义
- **核心职责**：[例如：负责弹塑性材料的本构积分，根据传入的应变增量更新应力与历史变量。]
- **绝对不负责**：[例如：绝对不负责单元刚度矩阵的组装；绝对不直接读取 L3 用户配置。]

## 2. 全景套件 v3.0 物理文件映射
| 逻辑件 (套件v3) | 当前落地文件 / TYPE / 过程 | 状态 (ACTIVE/WIP/TODO) |
|-----------------|----------------------------|------------------------|
| Def / Schema    | `MD_Mat_Def.f90` (Desc/State/Algo/Ctx) | ACTIVE |
| Main Algorithm  | `MD_MatElastic.f90`        | ACTIVE |
| Bridge          | `MD_MatRT_Brg.f90`         | ACTIVE |
| SIO Proc        | `MD_Mat_Proc.f90`          | ACTIVE |

## 3. SIO 接口与参数契约
- **标准入口**：`SUBROUTINE RT_Mat_Core_Eval(desc, state, algo, ctx, args)`
- **`_Arg` 封装要求**：必须使用 `MD_Mat_Arg` 封装附加参数，禁用散装参数。
- **状态流转**：
  - `[IN]`：`desc`, `algo`
  - `[INOUT]`：`state`, `ctx`

## 4. 四链与依赖边界
- **理论链**：引用《非线性有限元本构理论》第 X 章。
- **上行依赖**：依赖 L3_MD 的初始 Populate。
- **下行依赖**：被 L4_PH 的物理积分点循环调用。
- **禁止依赖**：禁止 `USE L5_RT`、禁止包含 MPI 指令。
```

---

## 3. 合同检验点 (Checklist)
1. 文件名必须精确为 `CONTRACT.md`。
2. 是否明确列出了“绝对不负责”的反向边界（这比正向职责更重要）？
3. `全景套件映射表` 中是否清晰体现了哪些物理文件承载了对应的逻辑职责？
