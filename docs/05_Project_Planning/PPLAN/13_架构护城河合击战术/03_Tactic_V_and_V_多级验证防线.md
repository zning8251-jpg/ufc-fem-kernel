# 战术三：V&V多级白盒测试与演武闭环 (演武)

> **文档位置**：`docs/05_Project_Planning/PPLAN/13_架构护城河合击战术/03_Tactic_V_and_V_多级验证防线.md`

## 1. 战术意图

Verification & Validation (验证与确认) 是严肃工业软件的生命线。既然 UFC 实现了“十件套”物理拆分，最大的红利就是**“每一块代码都变得可以被独立测试了”**！我们不再需要跑一个几万节点的完整 INP 才能知道其中某一个形函数是不是写错了。这是我们的“演武场”。

## 2. 核心武器一：纯函数单元测试 (Unit Mocking)

由于所有的 `_Algo.f90` 都被设计成了 `PURE`，输入只有 `Desc` 和 `State`。
- **实施策略**：我们可以直接用一套极简的 Fortran 框架（甚至不需要读取文件），在内存中直接硬编码伪造（Mock）一个 `Desc` 和一个 `State`，丢入 `_Algo`。
- **效果**：实现对于底层代数、张量分解、积分点权重的毫秒级测试。

## 3. 核心武器二：斑块测试与 MMS 验证矩阵 (Patch Test & MMS)

单单元测试只能证明“语法与数学公式没写错”，而**斑块测试 (Patch Test)** 与 **MMS (Method of Manufactured Solutions)** 才能证明“物理对不对”。

- **机制**：在 `L4_PH` (Element) 与 `L3_MD` (Material) 的联调层建立一个白盒自动化测试套件。
- 给定已知解析解的位移场（如纯剪切、纯弯曲），反向加载给组装好的 2x2 单元阵列。
- 测试其产出的节点反力（Residual）是否在 `1E-12` 级别归零，以验证全局形函数、体积刚度、剪切锁死的控制情况。

## 4. 落地路径

1. 所有核心域必须提交对应的 `_Test.f90`。
2. 将斑块测试和简单拉伸测试（1个六面体单元）集成到每一次 PR (Pull Request) 的冒烟测试中。
3. 只有通过了 V&V 多级验证的代码，才允许合并回 Master（对应 `TST-001` 规范）。
