# 架构与生态融合：双轨制API与用户扩展机制设计

## 1. 商业软件底层架构的“错觉”与真相
许多工程师在接触有限元内核开发时，往往会产生一种“架构错觉”：认为商业巨头（如 ABAQUS）内部的核算代码也是按照 UMAT/UEL 那种散装参数（如几十个扁平的一维/二维数组）的接口格式编写的。

**真相**：
ABAQUS 的内部核心早就重构为 C++ 和现代 Fortran 的混合体，材料和单元是极其复杂的对象（包含内存池引用、多场耦合指针等）。对外的 UMAT 接口只是为了**向后兼容（Backward Compatibility）**而保留的“上世纪遗物”。在内部引擎与外部插件之间，存在一个**拦截适配层（Adapter / Sandbox Barrier）**，负责在复杂的内部对象和扁平的外部数组之间进行打包（Pack）与解包（Unpack）。

## 2. UFC 的终极解法：双轨制与“特洛伊木马”战略
为了既保持 UFC 内核的绝对统一与优雅，又能无缝吸收 ABAQUS 庞大的生态遗产，UFC 在架构上采用**“双轨制 API”战略**：

### 轨道一：UFC 原生用户接口 (Native API)
*   **定位**：贯彻大统一原则，面向新生态与 AI 加速。
*   **形式**：强制使用 UFC 的结构化五参/六参门面（如 `CALL UFC_UMAT(desc, state, algo, ctx, args)`）。
*   **优势**：极度干净、类型安全，可直接享受 UFC 的内存池和 AI 零拷贝加速。

### 轨道二：生态兼容接口 (Legacy API)
*   **定位**：通过 `_Ext.f90` 建立“特洛伊木马”，白嫖现有生态。
*   **形式**：在 `_Ext.f90` 中提供兼容开关。当用户提供原始的 ABAQUS `UMAT.f` 时，`_Ext.f90` 作为“翻译官”，将 UFC 内部的优雅结构体降维拆解为散装数组，调用用户代码，进行 `NaN` 检疫后，再重新组装回 UFC 的 state。
*   **优势**：在保持内部主板“纯洁性”的同时，提供了完美的生态转接插口。

## 3. 用户扩展的数据槽位：为什么 CAE 不能使用“任意结构体”？
在现代软件工程中常见的“泛型编程与不透明指针（Opaque Pointer / User Data Slot）”，在 CAE 内核中是灾难，原因如下：
1.  **内存池与 GPU 卸载破产**：未知的黑盒对象大小不确定，导致无法在 `Init` 阶段一次性分配连续内存池，产生内存碎片，且 GPU 无法搬运未知大小的对象。
2.  **断点重启动（Checkpoint/Restart）瘫痪**：含有指针或链表的黑盒结构体无法被安全序列化写入 HDF5。
3.  **MPI 幽灵同步失败**：未知字节数的结构体无法通过网络边界进行高效通信。

### 正确解法：连续数组盲区 (STATEV / PROPS)
*   **机制**：用户在 L3 提前申报所需的空间大小（如 15 个 PROPS，50 个 STATEV）。
*   **执行**：UFC 在 `Init` 阶段分配巨大的、连续的浮点数组池。在 L4 求解时，将数组片段（Slice）移交给用户子程序。
*   **优势**：物理上保证了内存连续、MPI 通信和序列化安全；逻辑上，用户可以在自己的子程序中将这串数字“强行解释”为所需的任何复杂内变量，兼顾了极致性能与绝对的自由度。
