# UFC 核心闭环：域级十件套与合同卡落地总纲

> **v2.0 口径提示（2026-04-25）**：本文保留早期“十件套”物理后缀示例（如 `_Algo.f90`、`_Desc.f90`）作为历史语境。当前代码命名已升级为 **层级-域级-功能三段式**，主模块默认无 `_Ops` / `_Algo` 后缀；`Desc/State/Algo/Ctx` 优先作为 TYPE 后缀与逻辑职责。新版中间层口径见 [`05_中间架构层新版总纲_全景套件v3.0.md`](05_中间架构层新版总纲_全景套件v3.0.md)。

## 1. 设立初衷

随着 UFC 新核“顶层分析步驱动 -> 结构化降维积分 -> 弹塑性本构回归 -> 全局组装与求解”的核心物理闭环（任督二脉）彻底打通，我们需要将这一突破性的成果固化。

为了防止后续代码迭代再次引入高耦合的“God Object”或产生架构腐化，本专项旨在对闭环链路上的**每一个核心域（Domain）/子域（Sub-domain）**，严格套用**“域级合同卡 (Domain Contract)”**和**“十件套 (The Ten-Piece Set)”**的标准范式进行可行性设计与物理落地。

## 2. “十件套”架构标准模型（核心职责最小集）

任何一个标准化的业务子域（例如 `PH_Elem`、`MD_Mat`、`RT_StepDriver`），都必须被严格拆解为以下 10 个物理或逻辑分离的模块，以此保证职责的单一性与数据的纯洁性：

| 序号 | 模块后缀 | 模块职责与定位 | 状态属性 |
|---|---|---|---|
| 1 | `_Def.f90` | **静态定义**：全局常量、枚举值、基础静态结构体。 | 无状态 |
| 2 | `_Desc.f90` | **只读描述**：模型的只读属性（如材料弹性模量、单元节点坐标基准）。 | **纯只读** |
| 3 | `_State.f90` | **场与状态**：随着增量步推演而演化的场变量（如位移、塑性应变、内变量）。 | 读/写 (跨步演化) |
| 4 | `_Ctx.f90` | **运行时上下文**：单次迭代/计算过程中需要分配的临时内存（如单元刚度矩阵、临时雅可比）。 | 读/写 (步内清空) |
| 5 | `_Algo.f90` | **纯函数算法**：核心数学与力学计算模块，严格无副作用（如形函数计算、径向回归）。 | **无状态纯函数** |
| 6 | `_Brg.f90` | **防腐桥接/门面**：本域对其他层暴露的唯一合法入口，**承载合同卡契约**。 | 流程调度 |
| 7 | `_Reg.f90` | **工厂与注册表**：处理 ID 到具体算法的路由与分发（如 Element Family 路由）。 | 全局单例 |
| 8 | `_Err.f90` | **异常与错误码**：定义该域专有的错误码与异常处理逻辑。 | 静态只读 |
| 9 | `_Util.f90` | **局部工具**：该域专用的杂项与帮助函数。 | 无副作用 |
| 10| `_Test.f90` | **单元测试**：针对该域内 `_Algo` 逻辑的白盒测试套件。 | 测试专用 |

*(注：视子域复杂度，_Err, _Util, _Test 等辅助件可选择性物理合并，但逻辑必须分离)*

## 3. 域级合同卡 (Domain Contract Card) 机制

**域级合同卡**是 `_Brg.f90` 对外暴露的通信协议，它强制要求：
1. **禁止越级透传**：调用方不得将自身的巨型上下文指针（如旧版的 `Job` 对象）直接传给下层。
2. **结构体分层切片**：所有入参必须被严格切分为：配置项 (`in: Desc`)、状态项 (`inout: State`)、上下文 (`inout: Ctx`)。

### 合同卡标准模板：
```yaml
# 域级合同卡：[所属层]_[子域名称]
# 职责说明：一句话描述该接口的核心业务
Interface: [层级]_[子域]_Compute
Inputs:
  - [Desc_Type]   (INTENT IN)    : 静态属性/几何尺寸/材料常量
  - [State_Type]  (INTENT INOUT) : 当前增量步的场变量/历史变量
  - [Ctx_Type]    (INTENT INOUT) : 临时分配的工作内存与输出载体
Outputs:
  - [Status_Type] (INTENT OUT)   : 统一错误状态码，接管所有异常
```

## 5. 十件套 v2.0：逻辑职责与当前文件命名映射

> 本节是 11 专项的正式升级口径，用于把旧十件套（后缀驱动）与当前命名规范（三段式驱动）接上。

### 5.1 基本结论

- **十件套定义职责，不强制十个固定后缀文件。**
- **文件命名遵循** `{层}_{域}_{功能}[角色后缀]`，主模块默认无 `_Ops` / `_Algo`。
- **四型 `Desc/State/Algo/Ctx` 优先作为 TYPE 后缀与数据职责**，不再要求独立 `*_Algo.f90`。
- **`_Proc` 保留**给 L5_RT / SIO / Harness 入口，不做去后缀化。

### 5.2 逻辑件到物理命名映射

| 逻辑件 | 当前推荐落地 | 说明 |
|---|---|---|
| Contract | `CONTRACT.md` | 域职责、接口、SIO 偏好、依赖、迁移状态 |
| Def / 四型定义 | `*_Def.f90` | 常量、枚举、`*_Desc/State/Algo/Ctx` TYPE |
| Main Algorithm | `{层}_{域}_{功能}.f90` | 主计算/主流程模块，默认无 `_Ops` / `_Algo` |
| Bridge / Facade | `*_Brg.f90` | 跨层防腐与上下文切片 |
| Proc / SIO | `*_Proc.f90` | L5/SIO 统一入口 |
| Registry / Index / Map | `*_Reg.f90` / `*_Idx.f90` / `*_Map.f90` | 路由、索引、映射类角色 |
| Diagnostics / API | `*_Diag.f90` / `*_API.f90` 或主模块内聚 | 观测与外部接口按域复杂度决定 |
| Test / Mock | `Tests/` 或测试专用目录 | 不进入生产热路径 |

### 5.3 与旧十件套的解释关系

- 旧文中的 `XX_Algo.f90` 现在解释为“**算法职责所在模块**”，其物理文件常为 `XX.f90`。
- 旧文中的 `XX_Desc.f90` / `XX_State.f90` 常被收敛到 `XX_Def.f90`（承载四型 TYPE）。
- 如需拆分文件，可拆；如职责轻量，可合并；验收看 **职责完备 + 合同可验证**，不看“后缀数量”。

### 5.4 最小裁决规则（本专项执行）

1. **十件套负责职责完整性**：Contract、Def、四型、主算法、Bridge、Proc、Reg/Idx/Map、测试等职责可追踪。
2. **三段式负责文件命名**：主模块 `{层}_{域}_{功能}.f90`；角色明确时使用 `_Def/_Brg/_Proc/_Reg/_Idx/...`。
3. **四型负责 TYPE 命名**：`*_Desc` / `*_State` / `*_Algo` / `*_Ctx` 仅作为 TYPE 后缀。
4. **`CONTRACT.md` 负责代码落点**：每个域在合同卡中列出十件套 v2.0 映射与迁移状态。

> **最终执行策略（本专项）**：按“**十件套（职责） + 合同卡（边界） + 命名/依赖治理（约束）**”三件套推进。
