# UFC 全生命周期架构：接口优先级、全链路数据流与三级内存体系

在彻底打通了“计算核心引擎”之后，您提出的四个补充要求，将视线从“发动机内部”拉高到了“整车系统工程”。

有限元软件的死穴不仅在计算，往往在**接口腐化**、**解析/输出瓶颈**和**内存饥饿**。以下是针对这四大盲区的终极设计规范。

---

## 1. 接口签名的 T0 级最高绝对优先级（防腐化机制）

**【危机背景】**：
Fortran 开发者极易陷入“参数列表无限膨胀”的惯性（如早期 UMAT 有 30 多个参数）。即使有规范，在写新模块时，为了图省事传个临时变量，随时可能破坏 5参/6参 SIO 签名。

**【T0 级防御：结构化 IO（SIO）强制履约】**：
从现在起，**“SIO 签名替代扁平臃肿参数”被列为架构 T0 级最高优先级铁律**。

*   **代码拦截**：任何核心子程序（`_Core`, `_Proc`, `_Brg`）的接口，**严禁出现超过 6 个参数**。
*   **统一载体**：如果在计算中突然发现需要传入一个临时温度或输出一个摩擦耗散能，**绝对禁止在接口末尾追加参数**。必须去 `_Def.f90` 中，将这个新变量加入到 `Ctx`（输入）或 `Args`（输出）中。
*   **开发心智**：开发者必须像写 Web API 的 JSON Payload 一样写 Fortran 接口。接口只有 `Request (desc, state, algo, ctx)` 和 `Response (args)`。

## 2. 全生命周期数据流转（非计算环节的补全）

除了迭代计算，UFC 的整体数据管线（Data Pipeline）必须包含五个阶段：**Input $\to$ Parse $\to$ Compute $\to$ Map $\to$ Export**。

### 2.1 输入与解析（Input & Parse）：关键字域的降维打击
*   **功能**：读取 `.inp` 文本文件，转化为底层的内存池。
*   **设计**：建立专门的 **KeyWord 关键字解析域**。采用抽象语法树（AST）理念。
*   **链路**：文本行 $\to$ 分词（Lexer） $\to$ 关键字树（AST） $\to$ **反向注入（Reverse Populate）** $\to$ 生成 `L3_MD` 中的 `Desc` 基因池和拓扑连接。
*   **红线**：解析层绝对不能触碰计算层。解析完毕后，原始字符串数据必须被彻底丢弃。

### 2.2 结果映射与外推（Map / Writeback）：降噪与平滑
*   **功能**：计算引擎输出的原始数据往往在积分点上（如应力），且呈不连续跳跃。
*   **设计**：这是 `_Proc_[X][Y]Map` 的专属工作。
*   **链路**：在增量步收敛（`IncCmt`）后，调用形函数的逆矩阵，将积分点数据外推至节点。
*   **回传（Writeback）**：将外推平滑后的节点平均应力，作为历史状态刷回 `g_ufc_global` 中专门的后处理内存池。

### 2.3 结果输出（Export）：异步 I/O 隔离
*   **功能**：将收敛后的内存状态落盘。
*   **设计**：建立专门的 **Output 输出域**。
*   **链路**：由于磁盘 I/O 极慢，输出必须采用**缓冲池（Buffer Pool）**和**并行 I/O 框架（如 HDF5 或 MPI-IO）**。核心计算线程只需把数据推入缓冲池即可继续算下一时间步，由专门的后台线程（或单独的 I/O 步骤）负责落盘。

## 3. 三级内存架构体系（外存 - 内存池 - 缓存）

您提到的“按温度分类的 7 个内存池 + 3 个专属内存池”是非常先进的数据分层思想。为了配合极速计算，必须建立严密的三级硬件映射：

### 3.1 一级：外存（Out-of-Core Storage / Disk）
*   **设计职责**：当系统矩阵 $\hat{K}$ 达到几百 GB 超过物理内存时，或者为了重启（Restart）和后处理。
*   **数据形态**：分块的二进制文件、HDF5 数据库、`.res` 重启文件。
*   **交互逻辑**：仅在 `StepIni`（内存加载）和 `IncCmt`（状态落盘）时发生交互。

### 3.2 二级：全局内存池（Memory Pools / RAM）
*   **设计职责**：这就是 `g_ufc_global` 里的数据。即您说的“冷热数据分类”的存储区。
*   **热度分类**：
    *   **冷数据（Cold）**：如节点坐标、单元拓扑连接（`Desc`）。几乎只读，按 SOA（数组结构体）存储，占满内存。
    *   **温数据（Warm）**：历史状态（`State`），每个增量步读写一次。
    *   **专属池**：如 CSR 矩阵池、接触检索树专用池。
*   **交互逻辑**：全局持有，由 MPI 或 OpenMP 共享。

### 3.3 三级：缓存映射与软缓存（CPU Cache / `Ctx` & `Args`）
*   **核心痛点**：CPU 算得太快（纳秒级），去内存池取数据太慢（百纳秒级，Cache Miss 惩罚）。
*   **统一缓存设计（Software Cache）**：
    这就是为什么我们极力推崇 `Ctx` 和 `Args` 的根本原因！
    *   **`Ctx` 和 `Args` 就是为了填满 CPU L1/L2 Cache 而设计的“软缓存块”！**
    *   在 `Populate` 动作中，程序从庞大的内存池（RAM）中，把**当前这一个单元/积分点**所需要的数据剥离出来，塞进不到 1KB 的 `Ctx` 结构体中。
    *   这个 1KB 的结构体会被立刻加载到 CPU L1 Cache 里。随后交给 `_Core` 进行上千次的浮点运算。算完后结果装在 `Args` 里，再通过 `Assemble` 写回内存池。
*   **细粒度把控**：`Ctx` 的大小必须极度精简，必须保证内存地址连续。这也是为什么 L4 绝对不能包含 L3 数据的原因，因为一旦结构体过大溢出 L1 缓存，性能直接腰斩。

## 4. 最后一块短板：全局统一异常与中断体系（Exception Handling）

除了上述全生命周期，还有一个极其致命的空白——**如果算错了怎么办？**

Fortran 是没有 `try...catch...` 机制的。如果在几百层深的 `MD_Mat_Core_Evl` 中发生了除以零、负雅可比，或者材料软化破裂，如果程序直接执行 `STOP`，整个超算集群的心跳将瞬间死亡，所有未保存的数据全部丢失。

**【架构补全：异常冒泡（Error Bubbling）】**
1.  **全局错误字典**：在 `L2_NM` 或 `L1_IF` 中定义全局整数错误码（如 `UFC_ERR_NEGATIVE_JACOBIAN = -104`）。
2.  **统一载体**：在所有域的 `_Arg`（统一输出包）的头部，**强制包含两个变量**：
    ```fortran
    INTEGER :: err_code
    CHARACTER(len=64) :: err_msg
    ```
3.  **冒泡机制**：底层的 `_Core` 发生错误时，设置 `args%err_code`，然后立刻 `RETURN`。`_Proc` 捕获到非零代码后，停止组装，层层 `RETURN` 冒泡到 `L5_RT`。
4.  **顶层决断**：`L5_RT` 根据错误码决定：是切割步长（Cutback）重新试探，还是优雅地结束分析步并写入崩溃前的云图以便用户排查。
