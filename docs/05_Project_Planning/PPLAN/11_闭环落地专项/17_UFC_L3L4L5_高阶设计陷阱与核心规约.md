# L3/L4/L5 核心层高阶设计陷阱与架构规约

## 1. 架构命名校准：后缀归尾铁律
首先，正式确认并吸收架构师的命名纠正：**`PH_Elem_Desc`** 与 **`MD_Mat_Slave_Desc`**。
这不是“小问题”，这是关乎编译器解析和代码搜索一致性的“大问题”。
*   **纠正逻辑**：`Desc`, `State`, `Ctx`, `Algo` 作为架构的四大基石类型标识，**必须永远位于标识符的绝对尾部**。
*   **规范更新**：所有主辅 TYPE 的命名公式正式确立为 `[Prefix]_[Domain]_[Feature]_[SubFeature/Slave]_[TypeSuffix]`。

---

在理顺了命名、三维时空字典、以及扁平化数据真源之后，针对 **L3_MD（材料数据域）、L4_PH（物理单元域）、L5_RT（求解运行域）** 的内部引擎设计，还有以下几个极其致命的“深水区”需要特殊防范。

## 2. L3_MD (材料与模型数据层) 设计防坑点

### 2.1 维度坍缩与 Voigt 映射（彻底告别“一律 3D 嵌入”）
很多旧代码（包括此前的过渡版本）为了省事，不论是 2D 平面应力（CPS4）还是 3D 块体（C3D8），传给材料域的都是 6 分量（3D）应变。这在平面应力（Plane Stress）下会引发严重的厚度方向应变耦合错误。
*   **高阶设计**：`MD_Mat_Ctx` 必须包含精确的 `ndim` (空间维数) 和 `nshr` (剪切分量数)。
*   **规约**：对于 2D CPS4，`Ctx` 必须按 `ndim=2, nshr=1` (即 3 分量 Voigt 格式: $\varepsilon_{11}, \varepsilon_{22}, \gamma_{12}$) 传递数据。L3_MD 内部的 `_Core` 引擎必须根据 `ndim/nshr` 动态自适应张量大小，严禁硬编码 `size=6`。

### 2.2 状态历史变量 (State Variables - SDV) 的时序污染
*   **高阶设计**：`MD_Mat_State` 在迭代中极其容易被污染。牛顿迭代（Iter）中，材料会基于 $t_n$ 的旧状态不断试探，一旦把试探的塑性应变直接写回 `State`，后续的迭代就会带着错误的历史越算越错。
*   **规约**：`_Proc_PtItrEvl` 绝对不能修改 `State`，它只能把新的试探状态输出到 `Args` 中。只有当 L5_RT 宣布时间步收敛，调用 `_Proc_PtIncCmtEvl` 时，才允许将 `Args` 中的收敛数据正式刷入 `State`。

## 3. L4_PH (单元物理域) 设计防坑点

### 3.1 局部到全局的 DOF 映射 (Gather/Scatter Mapping)
单元算出来的 $K_e$ 是局部矩阵（如 C3D8 是 $24 \times 24$），它怎么知道该拼到全局 $\hat{K}$（可能是百万乘百万）的什么位置？
*   **高阶设计**：L4_PH 不能直接操作全局大矩阵，它必须是“黑盒”的。
*   **规约**：`PH_Elem_Proc_...` 在输出 `Args` 时，除了吐出局部矩阵 `K_e`，**必须强制吐出一个一维整数数组 `eqn_map(:)`（方程编号映射表）**。L5_RT 拿着这个映射表去负责全局的组装（Assemble）。

### 3.2 积分点缓存 vs 即时计算 (Memory vs Compute Bound)
*   **高阶设计**：形函数导数 $\nabla N$、雅可比矩阵 $J$ 和 $B$ 矩阵，每个迭代步都要用。是存入 `PH_Elem_State` 还是每次重算？
*   **规约**：现代 CPU 的计算速度远大于内存带宽（Memory Wall）。除了非线性大变形（几何非线性需要更新构型），对于小变形问题，严禁将巨大的 $B$ 矩阵缓存到全局内存中。应在每次迭代时，通过 `_Core` 即时（On-the-fly）计算 $B$ 矩阵。这极大降低了 L3_MD / L4_PH 的内存占用。

## 4. L5_RT (求解运行域) 设计防坑点

### 4.1 组装竞争（Race Condition in Assembly）
当采用 OpenMP 进行多线程组装时，多个单元（L4_PH）可能同时试图把自己的 $K_e$ 累加到同一个节点自由度上，导致数据覆写冲突。
*   **高阶设计**：绝不能让 L4_PH 负责组装。
*   **规约**：必须在 L5_RT 中实现安全的组装模式。要么采用**图着色算法（Graph Coloring）**将单元分组（同组互不相邻，可并发），要么在底层大矩阵的累加函数中使用 `OMP ATOMIC`。

### 4.2 求解器策略与收敛判定解耦（如 Arc-Length / L-BFGS）
如您正在测试的自适应 NR / 弧长法（ARC-CW）或 L-BFGS，如果把算法逻辑和收敛判定写死在一起，一旦发生强软化或爆炸发散，程序直接崩溃。
*   **高阶设计**：采用**策略模式（Strategy Pattern）**。
*   **规约**：L5_RT 的 `_Proc` 必须将**“步长控制（Arc-Length 约束方程）”**、**“求解推进（NR 或 L-BFGS）”**和**“收敛准则（力/位移/能量 Tolerance）”**完全剥离为三个独立的辅助 TYPE（辅 Algo）。当发生爆炸发散时，平滑地触发 `_Proc_GlbIncRbkEvl` (全局增量步回滚)，并通知切割步长，而不是让主程序抛出致命异常。
