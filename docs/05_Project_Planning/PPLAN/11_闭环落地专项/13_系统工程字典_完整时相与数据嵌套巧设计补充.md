## 1. 完整版的“时空+动作”三维时相字典 (Q1)

之前的 `(Init, Predict, Iter, Commit, Rollback)` 仅仅是**微观迭代轴**上的动作。一个完整的有限元架构，必须拥有**宏观时间轴（Macro-Time）**和**空间作用域（Spatial-Scope）**。

我们必须建立一个**“三维正交坐标系”**，所有子程序和流程的命名，必须由这三个维度的词汇组合而成：

### 维度 A：宏观时间轴 (Macro-Time)
控制物理现象推进的绝对时间：
*   **`Job`**：全局任务级（整个计算分析的生命周期）。
*   **`Step`**：分析步级（如：施加重力步、施加温度步）。
*   **`Inc` (Increment)**：增量步级（将一个 Step 切分为多个加载小步）。

### 维度 B：微观迭代轴与动作 (Micro-Iter & Action)
控制非线性方程组收敛与状态机的流转：
*   **`Init`**：初始化（赋初值、分配内存）。
*   **`Predict` / `Trial`**：预估/试探（进入迭代前的初值猜测）。
*   **`Iter`**：牛顿迭代步（包含组装、求解、本构测算）。
*   **`Correct`**：状态修正（如塑性返回映射中的应力修正）。
*   **`Commit`**：收敛固化（$t_{n+1}$ 状态写入历史变量，准备进入下一个 Inc）。
*   **`Rollback`**：发散回滚（废弃当前试探状态，准备退步 Cutback）。
*   **`Finalize`**：析构与清理。

### 维度 C：空间作用域 (Spatial-Scope)
明确该动作发生在哪个物理空间级别：
*   **`Global`**：全局系统级（宏观总刚度矩阵、总残差向量）。
*   **`Local` / `Elem`**：局部/单元级（单元内部的局部矩阵 $K_e$、节点域）。
*   **`Pt` / `IntPt`**：积分点级（最微观的材料本构计算点）。

**🌟 命名公式与化学反应：**
`[Domain]_[Feature]_Proc_[Scope][Time][Action]`
*   *示例 1*：`PH_Elem_Proc_LocalIncInit` (单元域-局部-增量步-初始化)
*   *示例 2*：`MD_Mat_Proc_PtIterCorrect` (材料域-积分点-迭代步-状态修正)
*   *示例 3*：`RT_Solv_Proc_GlobalStepCommit` (求解域-全局-分析步-收敛固化)

## 2. 数据结构嵌套的“巧设计”：上下文感知的去冗余 (Q2)

您指出的 `Arg%F%F_Slave` 现象，在软件工程中被称为**“蓝精灵命名反模式 (Smurf Naming Anti-pattern)”**。在具有严格层级和命名空间的强类型系统里，这种冗余是极其丑陋且破坏可读性的。

我们正式确立 UFC 数据结构设计的**“上下文感知去前缀化（Context-Aware Prefix Stripping）”铁律**：

### 核心法则：父节点即上下文，子节点严禁复读
当数据结构发生嵌套（无论是主 TYPE 嵌套辅 TYPE，还是外层容器包含内层变量）时，**内层成员的命名必须假设调用者已经知道了外层是谁，绝对禁止重复父级的名称、前缀或概念。**

**❌ 错误示范（冗余灾难）：**
```fortran
TYPE :: MD_Mat_Desc
    INTEGER :: MatType               ! 错：已经在 MatDesc 里了，无需加 Mat
    REAL    :: MatYieldStress        ! 错：重复 Mat
    TYPE(MD_Slave) :: MatSlaveData   ! 错：父级已定，无需加 Mat 和 Data
END TYPE

! 调用时变成了极其丑陋的：
! desc%MatType
! desc%MatSlaveData%Slave_E
```

**✅ 黄金设计（UFC 标准）：**
```fortran
TYPE :: MD_Mat_Desc
    INTEGER :: type_id        ! 对：直接叫 type_id
    REAL    :: yield_stress   ! 对：直接描述物理量
    TYPE(MD_Slave) :: slave   ! 对：干净利落
END TYPE

! 辅 TYPE 内部同样去冗余
TYPE :: MD_Slave
    REAL :: E                 ! 不要叫 Slave_E
    REAL :: nu                ! 不要叫 Slave_nu
END TYPE

! 调用时如丝般顺滑、符合人类直觉：
! desc%type_id
! desc%slave%E
```

**扩展至 `_Arg` (SIO) 结构：**
如果 `Arg` 里有一个主分支叫 `flux`，其内部还有一个从属的辅助通量分支，调用路径必须是干净的：
👉 `args%flux%slave` （绝对不允许出现 `args%flux%flux_slave`）。

## 3. 还有哪些极度重要的高阶考量？ (Q3)

除了时间字典和数据去冗余，在中间架构层和底层模板的缝合处，我们**必须（Must）**在此时纳入以下三大考量，否则系统在超大规模计算时必将面临性能或安全瓶颈：

### 3.1 线程安全隔离（Thread-Local vs Shared）
由于未来的组装（Assemble）和本构求值（Evaluate）必然是大规模 OpenMP 并行的。
*   **红线设计**：`Desc`（描述）在所有线程中必须是 **Shared & Read-Only**（全局共享只读）；而 `Ctx`（上下文）和 `Args`（输入输出包）必须是 **Thread-Private**（每个线程独立在栈上分配）。
*   `_Core` 引擎被设计为无状态（Stateless），天然完美契合 OpenMP 线程安全。

### 3.2 内存布局哲学：AOS vs SOA (结构体数组 vs 数组结构体)
Fortran 的极限性能取决于内存缓存命中率（Cache Line）。
*   如果我们将所有的材料状态定义为 `TYPE(MD_MatState), ALLOCATABLE :: states(:)`（这叫 **AOS**，Array of Structures），在提取单个变量（如全场塑性应变）时会导致严重的内存跳跃，阻碍 SIMD 向量化加速。
*   **系统级考量**：在 `_Def.f90` 定义局部单点 TYPE 的同时，全局大容器（`g_ufc_global`）在存储历史场变量时，应优先考虑 **SOA**（Structure of Arrays，即 `states%plastic_strain(:)`）或 **Blocked-AOS**（按 128 个单元分块），并在 `Populate` 动作中完成 SOA 到 AOS 的单点数据灌注。

### 3.3 接口防腐与契约版本控制（Schema Versioning）
工业软件的生命周期长达数十年，数据结构（`Desc`, `State`）会不断演进。
*   **设计预留**：在所有的主 TYPE（如 `MD_Mat_Desc`）的头部，必须隐式强制包含一个 `INTEGER :: version_id`。
*   **逻辑**：当未来我们为某种材料增加了新参数，旧的 ODB 结果文件或旧的 `UMAT` 接口依然可以通过识别 `version_id`，走旧版本的解析路由，实现完美的向下兼容（Backward Compatibility）。
