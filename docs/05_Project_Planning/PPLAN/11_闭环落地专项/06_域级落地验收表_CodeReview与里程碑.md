## 1. 约束分级（软 / 硬）

| 级别 | 含义 | 示例 |
|------|------|------|
| **硬约束** | 不满足则 **不合并 / 里程碑不通过** | 越层循环依赖；新增 `*_Ops.f90` / `*_Algo.f90` 主文件；`CONTRACT.md` 与公开 API 矛盾且无修订说明；无错误传播；破坏 SIO 五参/六参边界（L5/Harness 约定域） |
| **软约束** | 允许合入但须 **登记技术债**（Issue/合同「迁移状态」栏） | 十件套职责暂由其它域代管；命名紧凑 Token 待登记；测试仅 smoke、缺回归题；观测/diag 未齐 |

**Harness / 工具链**：视为 **硬约束的执行面**——在里程碑定义中声明「必须通过」的 harness / 构建 / 命名检查脚本时，未通过即等同硬约束失败。未纳入里程碑门禁的脚本可作软约束或建议项。

## 2. 验收主轴（本页核心：A / A+ / B / C）

### A. 合同卡与十件套 v2.0（边界 + 职责）

| # | 检查项 | 通过标准 |
|---|--------|----------|
| A1 | 本域 `CONTRACT.md` 存在且为当前权威 | 路径在 `ufc_core/**/<域>/CONTRACT.md`（或项目约定的域根） |
| A2 | **十件套 v2.0 映射表** | 含「逻辑件 → 文件/TYPE/过程 → 状态（Active/Deferred/Merged）」；允许多逻辑件合一文件，但 **不得职责悬空** |
| A3 | 输入输出与生命周期 | Desc/State/Algo/Ctx（及 `*_Arg` 若适用）的 INTENT、只读/跨步/步内规则写明 |
| A4 | 四链占位与落点 + **因果说明** | **（1）四链**：理论 / 逻辑 / 计算 / 数据 **各至少一句** **本域可核对** 说明，且各能 **指到** 具体 TYPE、过程、`*_Proc` 或数据流片段（四链闭集，不增第五链枚举）。**（2）因果说明（推荐）**：用 2–3 句自然语言，或等价给出 trace 级 `(trigger; upstream)`（谁触发、上游层/域/合同字段）；解释关键 **State/Ctx** **为何** 变化；**禁止**在合同或 Harness 中新增「因果链」与四链并列（见 `UFC/docs/02_Developer_Guide/UFC_数据结构与结构体规范.md` §5.6、`UFC/skills/fem-kernel-observability/SKILL.md`）。 |
| A5 | SIO / `*_Arg`（本域偏好） | 与合同「### SIO」小节一致；禁止仅为 `status` 的薄 `Arg`（与仓库 AGENTS 规则一致） |
| A6 | 域际关系 | 写明 **上游/下游域**、依赖方向；子域从属或并列关系可查（避免隐式耦合）；**推荐**将下列「域关系子表」粘贴进 `CONTRACT.md` 的「域际关系」小节 |

#### A 轴扩展：层域坐标 · 内外边界 · RT 六参 · 横切 · 门禁（适域勾选；**L5 / SIO 边界域、跨层域建议全开**）

| # | 检查项 | 通过标准 |
|---|--------|----------|
| A7 | **层与域坐标** | 合同或 PR 写清本域 **Layer（如 L3_MD / L4_PH / L5_RT）+ Domain 路径 + 功能集/子域**；与 `ufc_core/**` 目录及域清单 **一致、可检索** |
| A8 | **内外边界与 Bridge** | **对内**：四型（Desc/State/Algo/Ctx）职责、只读/跨步/步内规则、热路径与真相源与 A3 一致。**对外**：公开 API / `CONTRACT.md` 与实现同版本。**跨层**：仅经 **Bridge（B）/ SIO（S）/ 合同 TYPE（T）** 闭集，无未备案的隐式越层 `USE`（与 A6 子表、[`UFC_数据结构与结构体规范.md`](../../../02_Developer_Guide/UFC_数据结构与结构体规范.md) §1.1、§5.5 一致） |
| A9 | **RT 六参（若适用）** | 凡本域含 L5 `*_Proc`、Harness 约定 SIO 入口或须注入步进书签者：写明 **五参 vs 六参** 取舍；六参时 **`RT_Com_Base_Ctx` 为框架注入**，与域 `_Ctx` **正交**（见 [`UFC_数据结构与结构体规范.md`](../../../02_Developer_Guide/UFC_数据结构与结构体规范.md) §4.4、§5.4）。**不适用** 于纯 L3 冷数据域时在 PR/合同标 **N/A** 及理由 |
| A10 | **横切契约与可观测性** | 若域涉及 **序列化/版本/稳定字段**：与数据契约及合同「迁移状态」一致（可互链 `fem-kernel-data-contract`）。**可观测**：必要计数/计时/诊断；因果信息仅作 **trace/诊断元数据**，`chain_hint` 仍落在 **四链闭集**（见 `fem-kernel-observability` 技能） |
| A11 | **验证与 Harness 门禁** | PR 或合同写明须通过的 **测试命令**（编译/单测/集成/smoke）及 **`python UFC/ufc_harness/run_harness.py …` / `uhc.py …`** 等门禁（与 **§0** 裁决顺序、**C 轴** 互链；重复处可写「见 C1–C4」但 **须有一条可执行命令清单**） |

#### 域关系子表（A6 · 固定模板）

**填写说明**

- **关联域**：与 `ufc_core` 目录或域清单一致的标识，例如 `L3_MD/Material`、`L5_RT/Assembly_Solver`。
- **相对本域**：一律从 **本域** 视角填写——**上游** = 本域依赖其输出；**下游** = 对方依赖本域输出。若存在双向数据流，拆成两行或于 **备注** 说明切分点与防环策略。
- **契约类型**：从下表 **闭集** 取值（可填代号如 `T+B`，或写全称）；越层默认应有 **B** 或 **S**，避免裸 **U** 直连巨型上下文。
- **子域**：一般不单独占行；在 **备注** 标 `父域：…` / `从属：…` 即可。

**契约类型闭集（`CONTRACT.md` 中可与「接触面」列连用）**

| 代号 | 含义 |
|------|------|
| **T** | 合同 TYPE（`Desc` / `State` / `Algo` / `Ctx` 或其显式切片） |
| **B** | Bridge（`*_Brg` 或门面过程；防腐与上下文切片） |
| **S** | SIO（`*_Arg` + 五参/六参 `*_Proc`；L5/Harness 边界） |
| **R** | Registry / 路由（`*_Reg`、`*_Idx`、工厂、ID→实现） |
| **U** | 模块 `USE` 直链（须同层或符合单向依赖；跨层须备注 **已审** 理由） |
| **X** | 扩展点（UEL/UMAT/插件注册；与扩展合同互链） |
| **E** | 外部边界（L6、API、文件、序列化；若适用） |

### B. 命名与依赖治理（可检索 + 单向）

| # | 检查项 | 通过标准 |
|---|--------|----------|
| B1 | 三段式可解析 | `MODULE`/主文件名可分解为 `{层}_{域}_{功能}[角色]`；禁止「层缀+单泛词」歧义两段式（见统一命名方案） |
| B2 | 四型仅为 TYPE | `*_Desc`/`*_State`/`*_Algo`/`*_Ctx` 作 TYPE；**不**作为新主模块文件后缀 |
| B3 | 角色后缀 | `_Def`/`_Brg`/`_Proc`/`_Reg`/`_Idx`/… 仅在有明确角色时使用；L5 SIO 入口保留 `_Proc` |
| B4 | 依赖方向 | 符合六层单向依赖；无新循环；跨层仅经 **Bridge / 合同类型**（无巨型上下文越级透传） |
| B5 | `USE` 真实性 | 无指向不存在模块的 `USE`；无「仅装饰」的未使用 `USE`（含测试） |

### C. 可验证性（证据链）

| # | 检查项 | 通过标准 |
|---|--------|----------|
| C1 | 自动化证据 | 本域相关 **测试或 harness** 在 CI/本地可复现执行（命令或目标写在 PR 或 `CONTRACT.md`） |
| C2 | 回归粒度 | 至少：**编译通过**；关键路径有 **单测/集成测/smoke** 之一（里程碑级可要求更高） |
| C3 | 热路径与数据切片 | 合同或 PR 说明：热路径未引入 L3 大对象树无切片下沉；与 Populate/Bridge 责任一致 |
| C4 | 错误与观测 | 错误沿链传播；必要时有诊断/计数/日志钩子或后续 Issue 编号（软约束可登记） |
