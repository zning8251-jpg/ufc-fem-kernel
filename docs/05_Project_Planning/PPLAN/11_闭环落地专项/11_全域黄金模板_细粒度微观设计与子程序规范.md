# 全域黄金模板：细粒度微观设计与子程序级规范

在明确了“按域分类构建模板体系”的宏观骨架后，我们必须进一步深入到**微观层面（Micro-Architecture）**。

一个真正的工业级“黄金模板”，绝不仅仅是划分出几个 `.f90` 文件，而是要在每个 `MODULE` 内部建立精密咬合的齿轮系统：**细粒度到文件后缀的严格分工、主辅 TYPE 的数据嵌套、时相与动作的生命周期编排，以及统一的子程序签名（SIO）**。

---

## 1. 功能模块后缀（`_Suffix.f90`）的绝对物理隔离与职责判定

在同一个域（Domain）和功能集（Feature）下，多个 `.f90` 文件共同协作。它们的后缀决定了它们能写什么、绝对不能写什么：

| 文件后缀 | 模块命名示例 | 核心承载功能与红线纪律 |
| :--- | :--- | :--- |
| **`_Def`** | `MD_Mat_Def` | **契约与数据中心**。唯一允许定义 `Desc`, `State`, `Ctx`, `Algo` 四大 `TYPE` 的地方。严禁包含任何物理计算或复杂的可执行代码（除了简单的 `Init/Free` 内存分配分配）。 |
| **`_Core`** | `MD_Mat_Core` | **纯粹的数学与物理引擎**。承接所有的张量运算、本构积分、形函数插值、矩阵组装。红线：**无状态（Stateless）**，不允许自己保存历史数据，所有计算必须依赖外部传入的 `State` 和 `Ctx`。 |
| **`_Proc`** | `MD_Mat_Proc` | **业务流程与编排中心**。负责驱动时相（Time-phase）流转，调用 `_Core` 中的算法。它是外部调用本域的统一入口（Facade）。 |
| **`_Ops`** | `MD_Mat_Ops` | **状态操作与生命周期**。负责数据结构的序列化、反序列化、状态插值（如从积分点向节点外推）、数据的 Commit 与 Rollback。 |
| **`_Brg`** | `MD_Mat_Brg` | **桥接与沙盒**。防腐层（Anti-Corruption Layer），专门用于将 UFC 标准的四型数据，打包转化为 `UMAT`、`VUMAT`、`UEL` 等外部旧资产所需的扁平数组格式。 |

## 2. Module 内部数据结构设计：主 TYPE 与辅 TYPE 的派生树

在 `_Def.f90` 内部，数据结构绝不是零散的变量，而是高度面向对象的**嵌套派生树**：

*   **主 TYPE（总容器）**：统一以前缀加四型结尾，如 `MD_Mat_Desc`。
*   **辅 TYPE（多态分支）**：针对不同场景的具体实现，如线弹性、弹塑性。

**示例（材料域）：**
```fortran
! 辅 TYPE：具体场景的数据
TYPE :: MD_Mat_Desc_Isotropic
    REAL(c_double) :: E, nu
END TYPE MD_Mat_Desc_Isotropic

TYPE :: MD_Mat_Desc_Plasticity
    REAL(c_double) :: yield_stress, hardening_modulus
END TYPE MD_Mat_Desc_Plasticity

! 主 TYPE：通过 TYPE(C_PTR) 或 CLASS(*) 实现多态与聚合
TYPE :: MD_Mat_Desc
    INTEGER(c_int) :: mat_type
    TYPE(MD_Mat_Desc_Isotropic) :: iso_props
    TYPE(MD_Mat_Desc_Plasticity) :: plas_props
END TYPE MD_Mat_Desc
```
**设计红线**：模块的接口（子程序入参）**永远只传递主 TYPE**。`_Core` 在内部根据 `mat_type` 路由到辅 TYPE。这样即使增加 100 种新材料，子程序签名也永远不需要改。

## 3. 算法实现流转：按“时相 (Phase) + 动作 (Action)”编排

模块内的子程序命名与算法流转，必须符合有限元时间步进引擎的生命周期。我们采用 **“时相 + 动作”** 的正交命名规范：

### 时相（Time-phase，表示在非线性迭代的哪一步）：
1.  **`Init`**：分析步/增量步开始前的初始化（如赋予初值）。
2.  **`Predict`**：预估阶段（如弹性试探步 trial step）。
3.  **`Iter` / `Correct`**：牛顿迭代/塑性修正阶段。
4.  **`Commit`**：当前增量步收敛，固化状态（$t_{n+1} \to t_n$），写入历史变量。
5.  **`Rollback`**：当前步发散，状态回滚（退步重算）。

### 动作（Action，表示具体的计算行为）：
1.  **`Evaluate`**：计算并求值（如积分计算应力）。
2.  **`Assemble`**：矩阵或向量的组装。
3.  **`Map`**：场变量映射与插值。

**组合命名示例（各大场景规范命名）：**
*   `SUBROUTINE MD_Mat_Core_IterEvaluate(...)` ：在迭代步中计算材料本构。
*   `SUBROUTINE PH_Elem_Core_Assemble(...)` ：物理域的刚度组装。
*   `SUBROUTINE RT_Solv_Proc_Commit(...)` ：求解域收敛后的状态固化流程。

## 4. 细粒度子程序签名（SIO 规范）与数据流转契约

为了彻底消灭长达几十个参数的“子程序面条代码”，UFC 强制采用 **“结构化 IO (SIO)”** 签名。每个核心子程序**只能有 5 个或 6 个参数**。

### 统一形参公式：`(desc, state, algo, ctx, [RT_Com_Ctx], args)`

```fortran
!> 理论链：计算弹塑性一致切线刚度与应力更新 (Return Mapping)
!> 逻辑链：RT_Assembly -> PH_Elem_Core -> MD_Mat_Core_IterEvaluate
SUBROUTINE MD_Mat_Core_IterEvaluate(desc, state, algo, ctx, args)
    ! 1. desc: [IN] 材料描述（如 E, nu, 屈服面参数）。绝对只读。
    TYPE(MD_Mat_Desc), INTENT(IN)    :: desc 

    ! 2. state: [INOUT] 材料历史状态（如旧应力、塑性等效拉伸）。随积分演化。
    TYPE(MD_MatState), INTENT(INOUT):: state 

    ! 3. algo: [IN] 算法控制参数（如最大迭代次数、容差）。绝对只读。
    TYPE(MD_MatAlgo), INTENT(IN)    :: algo 

    ! 4. ctx: [IN] 当前积分点的上下文（如形变梯度、应变增量）。本步的驱动条件。
    TYPE(MD_MatCtx),  INTENT(IN)    :: ctx 

    ! 5. args: [INOUT] 统一 IO bundle。所有的“返回结果”打包在这里！
    ! 包含：新应力 (stress_new)、一致切线刚度矩阵 (D_ep) 等。
    TYPE(MD_Mat_Arg), INTENT(INOUT) :: args 

    ! ... 核心张量与本构积分实现 ...
END SUBROUTINE MD_Mat_Core_IterEvaluate
```

### 数据流转红线：
1.  **`Desc` 与 `Algo` 永远是只读的（INTENT(IN)）**。它们在分析步开始前就被解析完成，相当于“基因”。
2.  **`Ctx` 是驱动引擎**。外界把 $t_{n+1}$ 的试探应变或位移增量放在 `Ctx` 里传进来。
3.  **`State` 是记忆簿**。它负责跨越增量步保存历史变量。
4.  **绝不使用 `inp/out` 孤立参数**。所有的临时输入和计算结果，全部塞进同域的 `_Arg` 打包结构体中，并在 `_Arg` 定义时用 `[IN]` / `[OUT]` 注释严格区分。

---

## 5. 总结：这才是工业级的模板体系

通过这种微观设计，UFC 的任何一个新功能模块（无论是加一种新材料，还是写一个新的单元）：
1.  **物理文件一眼分明**：找数据去 `_Def`，找公式去 `_Core`，找流程去 `_Proc`。
2.  **数据关系绝对清晰**：主 TYPE 兜底，辅 TYPE 分支。
3.  **时序动作永不混乱**：`Init` -> `Evaluate` -> `Commit`/`Rollback` 闭环。
4.  **接口签名千篇一律**：永远是清爽的 5 参/6 参 SIO，极大地降低了心智负担和联调成本。

这就是将 UFC 的架构蓝图，真正落地为每一行代码规范的终极契约。
