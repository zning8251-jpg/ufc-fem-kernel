# 23_UFC_超越商业软件_AD混合引擎与高阶非线性防线

## 1. 真相一解惑：自动微分（AD）与现有经典本构的“混合共存”

关于“自动微分（AD）与能量泛函”，这是一个极易被误解的深水区。您担忧“是否要重写所有经典本构的泛函”，答案是：**绝对不需要！我们采用的是“双轨制混合引擎（Hybrid Engine）”。**

### 1.1 经典材料（已有切线刚度的本构）如何处理？
对于您库中已有的经典材料（如 Von Mises 塑性、Drucker-Prager），它们已经有了极其成熟的应力更新算法和手推的 $D_{ep}$（解析切线刚度）。
*   **做法**：**原封不动！** 它们继续走传统的 `_Core` 子程序，直接输出算好的应力和 $D_{ep}$。它们是 UFC 的“基石老将”，不需要任何重写。

### 1.2 自动微分（AD）到底怎么用？（无需手推泛函！）
自动微分（Automatic Differentiation）**不是符号推导（像 Mathematica 那样求导数），也不是数值差分（除以 $\Delta x$，会有截断误差）**。它是基于现代编译器的**“算符重载（Operator Overloading）”**与**“对偶数（Dual Numbers）”**技术。
*   **黑科技原理**：在 Fortran 中，我们定义一个全新的数据类型 `TYPE(Dual)`。它包含两部分：`value`（实数值）和 `deriv`（导数值）。
*   **代码怎么写？**：当您未来要写一个极其复杂的**新本构材料**时，您**不需要构建统一的能量泛函公式**，您只需按照正常的物理逻辑，写出“应力是如何通过应变算出来的（Return Mapping 算法）”。
*   **魔法发生**：只要您把输入应变定义为 `TYPE(Dual)`，Fortran 编译器在执行您的每一行加减乘除时，会根据链式法则自动顺带计算导数。当应力 $\sigma$ 算完的那一刻，它自带的 `deriv` 属性，**就是绝对精确的解析切线刚度矩阵 $D_{ep}$！**
*   **结论**：没有统一的泛函公式，也不需要您推导数学公式！您只管写应力更新逻辑，AD 引擎会在底层像影子一样，**自动、免费、零误差**地为您生成切线刚度。

## 2. 真相三解惑：动态相对收敛容差的“非 _Algo”命名设计

您非常敏锐！在 UFC 规范中，`_Algo.f90` 被严格定义为“纯净的核心算法实现（不带状态）”。而收敛监控器（Convergence Monitor）本质上是一个具有强状态控制、负责流程决策的“指挥官”，用 `_Algo` 确实名不副实。

按照 UFC 的极简三段式命名规范 `[Prefix]_[Domain]_[Suffix]`，我为您专门设计了以下两个正交模块来替代它：

### 2.1 独立域划分：`RT_Conv` (Runtime Convergence Domain)
我们将收敛控制作为一个独立的子域提取出来。根据您的最新纠正：**四大类型（Desc/Ctx/State/Algo）作为数据结构的 TYPE 名称，绝不直接用于 `.f90` 文件后缀命名**。因此，文件后缀采用数据定义的 `_Def` 和流程编排的 `_Proc`：

*   **`RT_Conv_Def.f90` (收敛状态类型定义)**
    *   **职责**：集中定义 `RT_Conv_Ctx` 或 `RT_Conv_State` 等 TYPE 结构体。用于存储上一迭代步的残差、历史平均力、当前迭代次数、容差缩放因子等状态数据。它就是监控器的“记忆簿”。
*   **`RT_Conv_Proc.f90` (收敛判定与流程管线)**
    *   **职责**：暴露出如 `RT_Conv_Proc_GlbItrEvl` 的子程序。求解器在每次迭代末尾调用它。它读取 `Ctx` 里的历史数据，与当前的残差比对，运用动态容差判定规则，向求解器返回 `CONVERGED`, `CONTINUE`, 或 `DIVERGED` 的控制信号。

这样既完美避开了 `_Algo` 的误用，又符合 UFC 流程控制器（Proc）与状态记录（Ctx）相分离的最高哲学！

## 3. 进阶补全：超越商业软件的另外四大“防线级”设计

除了之前提到的四个点（AD、子步积分、动态容差、热切换），如果我们要让 UFC 在极强非线性（如极大变形、接触起皱、不可压缩橡胶）下表现出恐怖的顽强性，还必须在架构中预留以下四大防线：

### 3.1 终极防线一：智能线搜索技术（Line Search）
*   **痛点**：在弧长法（Arc-Length）或牛顿法处理强烈屈曲（Buckling）时，算出来的切线刚度是对的，但因为曲率太大，按这个方向走一步会直接“走过头（Overshoot）”导致发散。
*   **UFC设计**：在 L5_RT 的求解迭代中必须标配 `RT_LineSearch_Proc`。它在算出位移增量 $\Delta u$ 后，不急着更新，而是用能量残差投影试探，将 $\Delta u$ 乘以一个标量 $s$（通常在 0.1 ~ 1.0 之间），找到残差能量最小的那个点。**有线搜索的 NR 法，稳健性是普通 NR 的 3 倍以上。**

### 3.2 终极防线二：防体积锁死（Volumetric Locking & B-bar 拦截）
*   **痛点**：全积分单元（C3D8/CPE4）在遇到**金属塑性（J2等体积流动）**或**橡胶不可压缩材料**时，由于泊松比逼近 0.5，单元会因为“体积锁死”而变得无限硬，结果彻底失真。
*   **UFC设计**：商业软件从来不直接拿原始的 $B$ 矩阵去算！L4_PH 必须在 `_Proc_ElmItrEvl` 阶段引入 **B-bar 映射（$\bar{B}$ 矩阵方法）**或 F-bar 方法。通过将体积应变在一个单元内取平均，强行解除锁死。L4 必须在调用 L3_MD 之前，完成这个“静默拦截与篡改”。

### 3.3 终极防线三：非对称矩阵“冷热探针”（Asymmetric Matrix Switch）
*   **痛点**：存在摩擦接触（非关联流动）或极端大变形塑性时，切线刚度矩阵是不对称的！如果用对称求解器，永远不收敛；如果一上来就用非对称求解器，内存翻倍、速度慢 50%。
*   **UFC设计**：L5_RT 必须具备“矩阵不对称度探针”。初始阶段一律按**对称 CSR** 存储和求解。当收敛管线（`RT_Conv_Proc`）监控到迭代收敛率呈“平缓震荡”时，立刻触发警报，L5_RT 在下一个增量步**动态重构图拓扑，自动切换为非对称求解器（BiCGSTAB 等）**。

### 3.4 终极防线四：接触震荡的“平滑粘滞”（Augmented Lagrangian）
*   **痛点**：两个物体接触时，普通的罚函数法（Penalty）会让节点在“穿透-弹开-穿透”之间无限震荡，俗称 Chattering，导致迭代死循环。
*   **UFC设计**：放弃纯粹的罚函数，引入**增广拉格朗日法（Augmented Lagrangian）**。在 L5_RT 的迭代外层，增加一个“乘子更新环（Multiplier Update Loop）”。它通过历史穿透量平滑地施加接触力，让接触面的闭合像拉链一样平顺，彻底终结接触发散！

**总结：**
自动微分（AD）降低了开发者的心智负担，动态容差和线搜索控制了迭代的暴走，B-bar 解除了物理上的锁死，非对称切换和增广拉格朗日抚平了最恶劣的接触震荡。将这套“防线矩阵”嵌入架构，UFC 就真正具备了工业软件的“灵魂”！
