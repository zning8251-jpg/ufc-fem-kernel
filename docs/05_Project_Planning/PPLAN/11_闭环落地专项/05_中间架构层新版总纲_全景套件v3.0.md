## 1. 三层架构定位

UFC 架构文档与代码实现分为三层：

| 层级 | 代表文档/目录 | 职责 | 不负责 |
|------|---------------|------|--------|
| **顶层：架构总纲** | `UFC/docs/05_Project_Planning/PPLAN/01_架构总纲/UFC_架构设计总纲_深度整合版_v5.0.md` 及 `二十三项字典` | 定义六层、四型、四链、单向依赖、L3 真相来源、系统工程原则 | 不逐个域规定文件清单 |
| **中间层：全景套件与合同卡** | `UFC/docs/05_Project_Planning/PPLAN/11_闭环落地专项/`、`UFC/docs/05_Project_Planning/PPLAN/12_全栈域级落地矩阵/` | 把总纲原则转换为每个域可执行的职责拆分、接口合同、落地矩阵 | 不要求每个逻辑件都对应一个固定物理后缀文件 |
| **底层：代码实现** | `UFC/ufc_core/` | 以 `CONTRACT.md`、Fortran 模块和测试实现中间层合同 | 不反向定义顶层原则；实现偏差必须回写合同/总纲 |

**裁决顺序**：

1. 架构原则冲突：以架构总纲与 PPLAN 为准。
2. 文件/模块命名冲突：以 `UFC/REPORTS/UFC_统一命名方案_层级域级功能三级体系_v1.0.md` 为准。
3. 域内接口冲突：以该域 `CONTRACT.md` 为准；若合同落后于代码，应先修合同或在变更说明中同步修订。
4. 编译/真实调用链冲突：以 `ufc_core` 当前可编译实现为真源，并回写中间层文档。

## 2. 核心澄清：全景套件是“逻辑职责”，不是固定文件后缀清单

早期十件套文档使用 `_Algo.f90`、`_Desc.f90` 等作为物理文件名示例。当前命名规范已经升级为：

```text
MODULE / 文件名 = {层}_{域}_{功能}[{角色后缀}]
```

因此从 v2.0/v3.0 起：

- **全景套件（原十件套）** 表达域内必须覆盖的 **逻辑职责**。
- **文件名** 遵循层级-域级-功能三段式；主计算模块默认 **无 `_Ops` / `_Algo` 后缀**。
- **`Desc` / `State` / `Algo` / `Ctx`** 优先作为 **TYPE 后缀与数据职责**，不是默认文件后缀。
- **`_Def` / `_Brg` / `_Proc` / `_Reg` / `_Idx` / `_Ext` / `_Comm` 等** 仅在有明确角色时作为文件后缀保留。

## 3. 全景套件 v3.0：17 项逻辑职责与当前物理落地映射

在 v2.0 的 13 项基础逻辑件之上，v3.0 新增了对接“高阶系统工程与未来生态”的 4 个高级组件（14~17项）。

| 序号 | 逻辑件 | 必选性 | 当前推荐物理落地 | 说明 |
|------|--------|--------|------------------|------|
| 1 | **Contract** | 必选 | `CONTRACT.md` | 域职责、输入输出、SIO 偏好、跨层依赖、实现状态的合同卡 |
| 2 | **Definition / Schema** | 必选 | `*_Def.f90` | 常量、枚举、四型 TYPE、参数束 TYPE；禁止把主计算逻辑塞入定义模块 |
| 3 | **Description (Desc)** | 必选或合并 | TYPE `*_Desc`，通常在 `*_Def.f90` | 只读模型描述；初始化后冻结；不得在热路径被任意改写 |
| 4 | **State** | 必选或合并 | TYPE `*_State`，通常在 `*_Def.f90` | 跨步演化状态；需要明确 commit/revert 或写回规则 |
| 5 | **Algorithm Descriptor (Algo)** | 按需 | TYPE `*_Algo`，通常在 `*_Def.f90` | 算法选择、开关、容差、策略参数 |
| 6 | **Context (Ctx)** | 必选或按需 | TYPE `*_Ctx`，通常在 `*_Def.f90` 或 `*_Ctx.f90` | 步内/迭代内临时工作区；热路径应避免频繁分配 |
| 7 | **Main Algorithm / Kernel** | 必选 | `{层}_{域}_{功能}.f90` | 默认主模块无后缀，承载主要计算、查询或管理过程 |
| 8 | **Bridge / Facade** | 跨层必选 | `*_Brg.f90` | 层间防腐层；执行上下文切片、类型转换、合同适配 |
| 9 | **Runtime Procedure / SIO Entry** | L5/编排按需 | `*_Proc.f90` | L5_RT、Harness、SIO 五参/六参入口保留 `_Proc` |
| 10 | **Registry / Index / Map** | 路由按需 | `*_Reg.f90`、`*_Idx.f90` | 注册表、ID 路由、索引查询、关键字映射等 |
| 11 | **Populate / Builder / Import** | 灌入按需 | 主模块或 `_Map` / `_Reg` | 从 L3/L6 输入填充 Desc/State/Ctx |
| 12 | **Diagnostics / IO / Dump** | 观测按需 | 主模块、`*_Diag.f90`、`*_API.f90` | 诊断、调试输出、序列化、外部接口 |
| 13 | **Test / Mock** | 验证必选 | `Tests/`、`*_Mock` | 不进入生产热路径；测试命名可用专用集合 |
| **14** | **Extension & Sandbox Barrier** | 外接必选 | `*_Ext.f90` 或 `*_User.f90` | **[v3.0新增]** UEL/UMAT 等第三方黑盒插件隔离与 C-Interop 沙盒校验 |
| **15** | **Parallel Comm & Sync** | 并行必选 | `*_Comm.f90` 或 `*_Sync.f90` | **[v3.0新增]** MPI 幽灵节点同步与领域边界通信屏蔽层 |
| **16** | **AI Proxy & Batched Engine** | AI域按需 | `*_Proxy.f90` 或 `*_Batch.f90` | **[v3.0新增]** AI 推理批处理打包，张量零拷贝机制的专属互操作层 |
| **17** | **Iterator & Path-Controller** | 高级求解 | `*_Iter.f90` 或 `*_Ctrl.f90` | **[v3.0新增]** 存放弧长法(ARC)、L-BFGS 等脱离普通 NR 的高级迭代策略 |

> 注：判断标准不是“是否凑齐17个后缀文件”，而是 **职责是否完整、边界是否清楚、合同是否可验证**，以及是否有清晰的物理落点。

## 4. 合同卡 v3.0 模板

每个生产域至少应有 `CONTRACT.md`，并包含以下小节：

```markdown
# <层>/<域> CONTRACT

## 1. 域职责
- 本域负责什么
- 本域不负责什么

## 2. 全景套件 v3.0 映射
| 逻辑件 | 当前文件 / TYPE / 过程 | 状态 |
|--------|------------------------|------|
| Contract | CONTRACT.md | Active |
| Def / 四型 | XX_Def.f90: XX_Desc/State/Algo/Ctx | Active |
| Main Algorithm | XX.f90 | Active |
| Bridge | XX_Brg.f90 | 若跨层则必选 |
| Proc | XX_Proc.f90 | L5/SIO 按需 |
| Extension/Sandbox | XX_Ext.f90 | 按需对接外设 |
| Parallel/Sync | XX_Comm.f90 | HPC 同步 |
| AI Proxy | XX_Proxy.f90 | 数据驱动/批处理 |

## 3. 输入输出合同 (SIO / *_Arg)
- 输入与输出约束、生命周期
- 五参/六参边界
- 零拷贝互操作说明（如有）

## 4. 四链位置
- 理论链、逻辑链、计算链、数据链
```

## 5. 迁移与审查规则

### 5.1 新代码与未来生态规则
- 绝对禁止将 **外部接口 (UMAT/UEL)** 写进核心力学算法中，必须切分至 `_Ext.f90` 建立沙盒。
- 绝对禁止将 **MPI/OpenMP 通信指令** 散落于热循环内，必须抽离并集中于 `_Comm.f90`。
- 面向 AI 预测的求解或本构更新，必须通过 `_Proxy.f90` 批处理调用，禁止退化为单点循环以保证 GPU 卸载带宽。
- 高阶迭代法（ARC-CW 等）必须位于 `_Iter.f90`，保持 `RT_SolvLin.f90` 纯洁性。

### 5.2 Code Review 检查清单
- [ ] L4/L5 核心中是否存在越界的 `MPI_Send/Recv` 污染？
- [ ] 外部 UMAT 是否通过 `_Ext` 完成了 NaN 拦截与类型适配？
- [ ] 数组是否具备 `BIND(C)` 的内存连续性以便将来 AI 零拷贝读取？
- [ ] 是否新增了 `*_Ops.f90` 或 `*_Algo.f90`（需叫停并纠正命名）？

## 6. 结论

从基础十件套（v1.0）到逻辑重塑（v2.0），再到如今的全景套件（v3.0），中间层架构已经彻底打通了与《二十三项全景字典》的对接。

未来的 UFC 开发，无论面对多大规模的并行、多么奇特的外接模型，或多复杂的 AI 驱动，都能在这个 **17 个逻辑坑位**中找到自己唯一且正确的位置。
