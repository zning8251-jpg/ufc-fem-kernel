## 1. 核心目标与意图

在完成对**核心闭环域（StepDriver, Element, Material, Assembly, Solver）**的“十件套”架构固化后（参见 `11_闭环落地专项`），本方案旨在将**“任督二脉”的物理闭环思路全面推广至整个 UFC 六层架构（L1-L6）中的所有 55+ 核心子域**。

本总纲为全域提供统一的落地推演模板，确保整个 UFC 从底层基础设施到顶层应用，都具备一致的架构收敛性、单向依赖性，以及**物理隔离与无锁组装**的特性。

## 2. 什么是“十件套与合同卡”范式？

为了彻底消除 God Object 并将物理意图固化在代码目录中，每一个功能域都必须强制进行“结构化粉碎”，按以下“架构件”进行落地：

### 2.1 合同卡（Domain Contract）
每个域必须对外声明其唯一的、极其狭窄的数据依赖。
- **输入**：读什么切片数据？
- **输出**：写什么切片数据？
- **原则**：只读不可写（`INTENT(IN)`）和严格的内存对齐约束。

### 2.2 十件套模块分布（The Ten-Piece Set）
每个域（如 `L4_PH_Element`、`L2_NM_Linalg`）在逻辑职责上被分解为：
1. **`_Def` (Definition)**: 定义域级常数、枚举、参数字典。
2. **`_Desc` (Description)**: 定义问题描述、几何属性、只读参数（初始化后不可变）。
3. **`_State` (State)**: 定义迭代状态、历史变量（读写分离：`state_old`, `state_new`）。
4. **`_Ctx` (Context)**: 域级临时上下文（避免内存频繁分配）。
5. **`_Algo` (Algorithm)**: 纯函数无状态核心算法计算职责；当前推荐主算法文件使用 `{层}_{域}_{功能}.f90`，`*_Algo` 主要作为 TYPE 后缀或历史术语。
6. **`_Factory` / `_Populate` (Builder)**: 从上层数据结构灌入/反序列化生成本域的 Desc。
7. **`_Brg` (Bridge)**: 门面层，向外提供标准化调用，组装下层的 algo。
8. **`_Container` (Domain Container)**: L3_MD 及以上特有，管理全域对象池生命周期。
9. **`_IO` / `_Dump` (Serialization)**: 诊断调试和可视化读写模块。
10. **`_Test` / `_Mock` (Unit Testing)**: 面向本域的独立单元测试桩代码。

### 2.3 后续全栈推广规则（L1-L6 统一口径）

当十件套推广到 `L1_IF`~`L6_AP` 全域时，统一采用以下规则：

1. **职责先行**：十件套用于保证每个域职责完整，不要求“每件一个固定后缀文件”。
2. **命名归一**：MODULE/文件名遵循 `{层}_{域}_{功能}[角色]`，主模块默认无 `_Ops` / `_Algo`。
3. **四型归位**：`*_Desc` / `*_State` / `*_Algo` / `*_Ctx` 作为 TYPE 命名，不再强制对应 `*_Desc.f90` / `*_Algo.f90`。
4. **角色保留**：`_Def` / `_Brg` / `_Proc` / `_Reg` / `_Idx` 等角色后缀在确有职责时保留。
5. **合同卡落地**：每个域在 `CONTRACT.md` 中显式列出“十件套 v2.0 映射 + 文件名映射 + 迁移状态”。

该规则以 `11_闭环落地专项/05_中间架构层新版总纲_全景套件v3.0.md` 与 `REPORTS` 命名规范为共同裁决基线。

## 4. 全栈落地推演标准模板 (Standard Documentation Template)

在后续的每个 `XX_落地推演.md` 中，所有的域都必须使用以下模板进行严谨分析：

```markdown
### 域级编码： [例如：`NM_Tensor`]

#### 1. 核心意图与完成体目标
- **设计意图**：为什么需要这个域？（解决什么耦合/痛点）
- **完成体目标**：在全闭环体系下它的最终形态是什么？

#### 2. 合同卡设计 (Domain Contract)
- **输入依赖**：依赖什么上层/同层对象的哪些只读切片。
- **输出写回**：负责刷新哪些缓冲/状态内存。

#### 3. 十件套 v2.0 映射 (Logical-to-Physical Mapping)
| 模块类型 | 文件名推断 | 核心职责 |
| -------- | ---------- | -------- |
| _Def     | `XX_Def.f90` | ... |
| Desc / State / Algo / Ctx | TYPE `XX_Desc` / `XX_State` / `XX_Algo` / `XX_Ctx`（通常在 `XX_Def.f90`） | ... |
| Main Algorithm | `XX.f90` | ... |
| _Brg     | `XX_Brg.f90` | ... |

#### 4. 打通思路与底层代码推断
- **架构件交互逻辑**：如何防越权操作。
- **并发与热路径设计**：是否有 OpenMP 化设计。
- **伪代码推断**：骨架级伪代码，证明接口封闭性。
```
