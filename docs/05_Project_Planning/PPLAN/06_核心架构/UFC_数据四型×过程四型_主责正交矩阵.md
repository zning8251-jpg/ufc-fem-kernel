## 摘要（一页读懂）

1. **数据四型**（已有）：回答「袋里装什么」— **Desc / Ctx / State / Algo**。
2. **过程四型（辅轴、可选）**：仅在需要 **统一描述主数据流**（审计、registry、文档）时使用；凝缩名 **Cval / Kern / Redu / Drv** 见第一节。**日常开发**：先 **功能与子功能**，再 **步骤 1…n**，再落到 **子程序/SIO**——**不必**先套四型过程。
3. **主责矩阵（辅工具）**：若采用标注，则格子表示 **(ProcKind, DataKind)** 的 **主** 读/写/无责；**不是** `INTENT` 表，**不是**硬隔离；「—」≠ 禁止。
4. **Observe**：默认 **不** 引入第五型；日志/输出为主路径的**附带**标注；唯一目的为观测时 **主标 Drv（DRIVE）**，并标 `! Obs: primary`。
5. **与仓库其它轴**：**Layer** 管边界；**SIO / `*_Arg`** 管接口形态；**ProcKind**（若用）管 **事后/横切** 角色叙述——**从属于**功能与数据设计。
6. **`.f90` 专职后缀**：**28 项闭集**（分组 A–F）+ **G 节互斥择一**；**禁止**新建 **`_Desc` / `_Ctx` / `_State` / `_Algo`**（与四型 TYPE 字面撞车）。**`Proc` / `Drv` / `API` / `Idx` 等** 表示 **功能简称**——**优先**；若与下文 ProcKind 凝缩 **同拼写**，在 `CONTRACT.md` **一行消歧**（见零节）。新增后缀走 **RFC + 表增行**，软上限 **32**。

## 零、开发主轴（功能优先）与本文档定位

**正常开发顺序（推荐）**：

1. **划功能边界**：本 Module/域要解决什么问题，对外承诺什么能力。
2. **用数据四型组织「袋里有什么」**：哪些 **Desc**（配置）、**Ctx**（胶水）、**State**（演化）、**Algo**（算法描述）——先 **TYPE/字段**，再谈过程。
3. **写清计算流程**：子功能分解 → **第 1 步…第 n 步**（可画成序列：读入、装配、求解、更新…）。
4. **落实现体**：为每一步选择 **`SUBROUTINE`/`FUNCTION`**，并按 **SIO / `*_Arg`** 统一签名；文件名/模块名用 **`Proc`、`Drv`、`API`、`Idx`、`Ops`…** 这类 **功能简称**（见第一节闭集），**从功能出发命名**，而不是先贴「契/核/汇/控」。

**契‑核‑汇‑控 / Cval/Kern/Redu/Drv 的定位**：它们是 **事后归纳用的「主数据流透镜」**——便于审计、registry、跨团队对齐「谁主要写哪类 TYPE」。**不是**开工第一件事，**不**替代步骤设计与功能命名；若某子程序难以单点归类，**允许不标**或只标 **辅** 标签。

**与第一节表格的关系**：A–F 各表中的 **「主 ProcKind」** 列含义是：**若** 你在注释/registry 里 **选用** `@ProcKind`，则常见对应关系如此；**不是**「实现该功能必须先选 ProcKind」的清单。

## 一、双四型语义

### 1.0 命名谱系（与 Desc/Ctx/State/Algo 对仗）

**数据四型**始终对齐 **TYPE 设计**；**过程四型**对齐 **（可选的）主数据流叙述**——与 **功能简称**（`Proc`、`Drv`…）**并存**：前者管「读/写哪袋数据为主」，后者管「在仓库里叫什么、在步骤链里干哪一步」。

| 一字         | 凝缩码（推荐） | 全大写（枚举/日志列名） | 单字母      | 完整英文 |
| ------------ | -------------- | ----------------------- | ----------- | -------- |
| **契** | **Cval** | **CONTRACT**      | **C** | Contract |
| **核** | **Kern** | **KERNEL**        | **K** | Kernel   |
| **汇** | **Redu** | **REDUCE**        | **R** | Reduce   |
| **控** | **Drv**  | **DRIVE**         | **D** | Drive    |

**并列写法（与四型数据同一粒度）**：

- 数据四型：**`Desc / Ctx / State / Algo`**
- 过程四型：**`Cval / Kern / Redu / Drv`** — 与 **`CONTRACT / KERNEL / REDUCE / DRIVE`** 严格一一对应，可混用但勿换序改名。

**关于 `Cval`**：取 **C**ontract **val**idation（合法化/契约化）义，对应「契」；避免用 `Cont` 以免与 *continuation*、*container* 等常见缩语冲突。若某子系统已有 `CVAL` 其它含义，可在该子系统内改用文档别名 **`Ctrt`**（仍表 contract），但仓库级默认以 **Cval** 为准。

**Fortran 标识符**：凝缩码多用于文档、注释、`@ProcKind` 文本；源码枚举名可继续 `UF_PROCKIND_CONTRACT` 等全名，或与凝缩码并存时在 `CONTRACT.md` 中写明映射。

#### 凝缩码与「Module 内过程名」、`.f90` 文件名

| 问题                                                                | 结论                                                                                                                                                                                                                                                               |
| ------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Cval/Kern/Redu/Drv 是不是 Module 里子程序的「法定名字」？** | **不是。** 它们是 **ProcKind 角色标签**，与 `SUBROUTINE`/`FUNCTION` 的 **工程命名**（域、层、物理对象，如 `PH_Mat_Elas_Eval`）正交。过程名应优先表达 **做什么**；角色用注释 **`! @ProcKind: Kern`**（或 registry）表达即可。 |
| **可否把凝缩码嵌进过程名？**                                  | **可选。** 例如 `Foo_Init_Cval`：在强拆分、生成代码或单文件单角色时可用；默认 **不推荐** 每个过程都带后缀，以免名字冗长且与域前缀重复。                                                                                                              |
| **可否作 `.f90` 文件名后缀？**                              | **可以，但非默认规范。** 典型 `*_Ops.f90` 往往 **同时** 含 Cval（Init）、Kern（Eval）、Redu（Assemble）等多角色，用单一 `*_Kern.f90` 会 **假暗示** 文件内只有 Kern。更稳妥的是下一节的 **「文件解剖学」后缀**（与 ProcKind 正交）。    |
| **何时适合用 `*_Kern.f90` 这类后缀？**                      | 文件**几乎只含** 该 ProcKind（例如仅装配的 `PH_ElemC3D8_Redu.f90`），且团队在 `CONTRACT.md` 中写明约定时；或 **工具生成** 的按角色分文件输出。                                                                                                     |

**一句话**：**Cval/Kern/Redu/Drv 描述「这类子程序在矩阵里站哪一格」，不替代「子程序/文件在仓库里叫什么」。** 作 `.f90` 后缀是 **团队可选约定**，不是 UFC 全局强制。

### 后缀策略：默认无后缀（新）与 `_Ops` 存量双轨

#### 原则

UFC 采用 **方案 A 双轨制** 后缀策略，平衡语义清晰度与存量代码兼容性：

1. **新代码/新域推荐原则**：**默认主计算文件无后缀**；**`_Def` / `_Brg` / `_Idx` / `_Reg` / `_Map` / `_Mgr` / `_API` / `_Ctrl` / `_Proc`** 仅在有 **明确非默认角色** 时出现
2. **存量代码保留**：现有 `*_Ops.f90` 长期保留，不强制批量改名
3. **增量代码鼓励**：新域或新拆分文件鼓励使用无后缀格式（如 `PH_Mat_Elastic.f90`）
4. **域内约定**：各域可在 `CONTRACT.md` 中声明「本域主模块无 `_Ops` 后缀」

#### 命名对比

| 场景         | 存量命名                   | 新代码推荐                 | 说明                                  |
| ------------ | -------------------------- | -------------------------- | ------------------------------------- |
| 默认计算模块 | `PH_Mat_Elastic_Ops.f90` | `PH_Mat_Elastic.f90`     | 默认无后缀                            |
| 类型定义     | `PH_Mat_Elastic_Def.f90` | `PH_Mat_Elastic_Def.f90` | `_Def` 保留                         |
| 桥接模块     | `PH_BrgL3_Ops.f90`       | `PH_BrgL3.f90`           | `_Brg` 保留，去掉 `_Ops`          |
| 索引模块     | `PH_L4Idx_Brg.f90`       | `PH_L4Idx.f90`           | `_Idx` 保留，去掉 `_Brg`          |
| 注册模块     | `PH_ElemReg_Ops.f90`     | `PH_ElemReg.f90`         | `_Reg` 保留，去掉 `_Ops`          |
| SIO过程      | `RT_Solv_Proc.f90`       | `RT_Solv_Proc.f90`       | `_Proc` 保留（L5/SIO/Harness 形态） |

#### 专用后缀闭集（精简版）

| 后缀      | 用途        | 示例                                            |
| --------- | ----------- | ----------------------------------------------- |
| `_Def`  | TYPE定义    | `PH_Mat_Elastic_Def.f90`                      |
| `_Brg`  | 层间桥接    | `PH_BrgL3.f90`                                |
| `_Idx`  | 索引管理    | `PH_L4Idx.f90`                                |
| `_Reg`  | 注册管理    | `PH_ElemReg.f90`                              |
| `_Map`  | 映射转换    | `MD_KW_Map.f90`                               |
| `_Mgr`  | 管理器      | `IF_Device_Mgr.f90`                           |
| `_API`  | 外部接口    | `IF_StructFormat_API.f90`                     |
| `_Ctrl` | 控制逻辑    | `RT_ContCtrl.f90`（可选，优先用 basename）    |
| `_Proc` | SIO过程单元 | `RT_Solv_Proc.f90`（L5/SIO/Harness 形态必需） |

#### 复合角色处理

对于同时表达多个角色的文件（如 `PH_L4Idx_Brg` 同时表达「索引 + 桥」）：

- **优先方案**：合成到 basename，如 `PH_L4IdxBrg.f90`（无后缀）
- **备选方案**：保留信息量大的后缀，如 `PH_L4Idx.f90`（丢掉「桥」）
- **域内约定**：在 `CONTRACT.md` 中明确优先级规则

#### 迁移策略

1. **不强制改名**：存量 `*_Ops.f90` 可保留
2. **域内迁移**：在域级打通过程中逐步统一到新风格
3. **新代码优先**：新拆分文件优先使用无后缀格式
4. **工具支持**：命名检查器同时支持新旧两种格式

#### 与第一节闭集的关系

本节后缀策略（精简版9项）与第一节闭集（28项）的关系：

- **第一节闭集**：仓库实际后缀的完整清单，用于存量代码理解和工具链
- **本节策略**：新代码的推荐原则，用于简化命名和提升可读性
- **双轨并存**：存量遵循第一节闭集，增量遵循本节策略

#### `.f90` 文件名后缀：专用闭集（与 ProcKind 正交）

**用途**：后缀 **优先** 表达 **功能专职**（`Proc`、`Drv`、`API`…）；**不等价于** 契/核/汇/控标签——后者仅作 **可选用** 的 `@ProcKind` 叙述（零节）。**枚举数量可控**：本闭集 **登记 28 项**（不含禁用项；分组见 A–F）；远小于「子程序数」，足够编成清单、做命名检查与 CI 白名单。

**保留字（强烈不建议新建 `.f90` 后缀）**：**`_Desc` / `_Ctx` / `_State` / `_Algo`**——与 **DataKind 四字** 及 `TYPE(..._*)` 默认联想强绑定，易误读。**功能后缀优先**：`**Proc` / `Drv` / `API` / `Idx` / `Asm`…**（见 A–F 闭集）表达「做什么」。若 basename **恰巧** 与 **Cval/Kern/Redu/Drv** 同拼写，在 **`CONTRACT.md` 写清**：此处指 **功能模块名** 还是 **ProcKind 矩阵标签**（二者可共存，不应 silent 混用）。

**闭集治理**：新增后缀 = **RFC 一行理由 + 本表增行 +**（若有）**`naming_checker` 白名单更新**；超过 **32 项**应优先 **合并近义** 或 **下沉到子目录 basename**，避免后缀爆炸。

> **表内「主 ProcKind」列**：仅当采用 **`@ProcKind`** / registry **选用** 矩阵标注时，才有典型对应；**非**功能设计的必经列。

##### A. 定义与粘合（声明为主）

| 后缀          | 专用含义                                                                | 与近邻分界                                                                 | 主 ProcKind    |
| ------------- | ----------------------------------------------------------------------- | -------------------------------------------------------------------------- | -------------- |
| **Def** | `TYPE` / `ENUM` / `PARAMETER` 等 **纯声明**；尽量无可执行体 | 有薄过程但无域主逻辑 →**Glu**；可执行域逻辑 → **Ops**        | **Cval** |
| **Glu** | **纯粘合**：指针聚合、薄 accessor、`CONTAINS` 仅包装            | 纯声明 →**Def**；出现本构/单元主算 → **Ops** / **Loc** | **Cval** |

##### B. 计算与操作（可执行为主）

| 后缀          | 专用含义                                                                                                     | 与近邻分界                                                              | 主 ProcKind                            |
| ------------- | ------------------------------------------------------------------------------------------------------------ | ----------------------------------------------------------------------- | -------------------------------------- |
| **Ops** | **域内混合**：Populate 片段、本构、单元族、Init+Eval 同文件                                            | 几乎全局部 IP/单元算子 →**Loc**；几乎全全局累加 → **Glb** | **混**                           |
| **Loc** | **局部算子独占**：B 阵、形函数、IP 本构、单元 Ke/Re（**不写** `K_global+=`）                   | 一旦出现稳定的全局累加主路径 →**Glb** / **Asm**            | **Kern**                         |
| **Glb** | **全局归约独占**：`K+=Ke`、`R+=Re`、约束装配主路径（PH 侧或 RT 侧 **不含**「流水线外壳」时） | 需要 L5 装配**阶段机/管线** 组织 → **Asm**                 | **Redu**                         |
| **Asm** | **装配流水线**：阶段顺序、稀疏图侧与全局组装 **编排**（仍可有 Redu 内核）                        | 仅单次累加、无阶段语义 →**Glb**                                  | **Redu**（外壳可 **Drv**） |

##### C. 映射、注册与分派

| 后缀          | 专用含义                                                                 | 与近邻分界                                                        | 主 ProcKind                    |
| ------------- | ------------------------------------------------------------------------ | ----------------------------------------------------------------- | ------------------------------ |
| **Map** | **语义映射**：关键字/卡片字段 → **模型树/域对象**（建语义） | 只做「名字→内部 ID」、不建语义树 →**Sym**                 | **Cval** + **Drv** |
| **Sym** | **符号与名字空间**：符号表、重名决议、scope 栈                     | 与**Map**：Sym **不**负责把卡片变成物理模型           | **Cval**                 |
| **Reg** | **静态注册表**：材料表、单元族表、插件槽位 **元数据**        | 运行时 vtable/函数指针**解析** → **Dsp**             | **Cval** + **Drv** |
| **Dsp** | **动态分派**：vtable、函数指针、运行时 material point 路由         | 与**Reg**：Dsp **可执行**解析；Reg **偏静态表** | **Drv**                  |

##### D. 策略、求解、步进与快照

| 后缀            | 专用含义                                                                      | 与近邻分界                                                       | 主 ProcKind                                          |
| --------------- | ----------------------------------------------------------------------------- | ---------------------------------------------------------------- | ---------------------------------------------------- |
| **Strat** | **策略选择**：用哪种求解器/本构分支/分析路径的 **选项与分支面**   | 实现「解一次线性系统/牛顿步」内核 →**Solv**               | **Drv**                                        |
| **Solv**  | **求解内核**：迭代解、线搜索算子、PCG/Newton 等 **数值代数**      | 与**Strat**：Solv **不**负责「选策略卡片」           | **Kern** / **Redu**（视实现）            |
| **Step**  | **增量控制**：时间步、载荷增量、子步进 **控制流**                 | 与**Run**：Step **改怎么进**；Run **记进到哪** | **Drv**                                        |
| **Run**   | **运行快照袋**：当前步计数、阶段 ID、只读管线 token 等 **数据面** | 与**Step**：Run **不**承担分支策略                   | **Drv**（写 **DataKind::Ctx** 字段常见） |

##### E. 桥接、Populate、写回（跨边界）

| 后缀          | 专用含义                                                                | 与近邻分界                                                        | 主 ProcKind                    |
| ------------- | ----------------------------------------------------------------------- | ----------------------------------------------------------------- | ------------------------------ |
| **Brg** | **泛层桥**：L3↔L4、L4↔L5、任意跨层 **适配与数据形状变换** | 专写「结果回写」→**Wb**；专填 L4 Populate → **Pop** | **Drv** + 被调子集       |
| **Pop** | **L4 Populate 专页**：把 L3 模型 **灌入** PH 数据结构       | 与**Brg**：Pop **不**承担跨命名空间层的一般适配       | **Drv** / **Cval** |
| **Wb**  | **Write-back 专页**：解→模型/ODB/重启文件的 **回写**       | 与**Brg**：Wb **不写**通用跨层适配                    | **Drv**                  |

##### F. 运行时、工作区、环境与观测

| 后缀             | 专用含义                                                               | 与近邻分界                                                                                                   | 主 ProcKind                                                         |
| ---------------- | ---------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------- |
| **Proc**   | **L5 SIO 过程单元**：`_Proc.f90`、五参/六参 **形态锚点** | 与**Orc**：Proc 以 **SIO 签名** 为纲；Orc 为域内小编排                                           | **Drv** / **Redu**                                      |
| **Drv**    | **Driver（功能名）**：步进/作业/求解外壳、RT 主循环邻近逻辑      | 与**Orc/Step**：**Drv** 偏 **对外驱动面**；**Orc** 域内调度；**Step** 增量规则 | 常为矩阵**Drv**；**若与 ProcKind 标签同字，见零节消歧** |
| **Orc**    | **小编排壳**：单域内调度子例程（无 SIO 硬约束）                  | 与**Brg**：Orc **不跨层**；Brg **可跨层**                                                  | **Drv**                                                       |
| **Wsp**    | **Workspace**：scratch、缓冲池、workspace 槽位选择               | 与**Env**：Wsp **算内存储**；Env **算外执行环境**                                          | **Drv** + **Redu**                                      |
| **Env**    | **Environment**：MPI 通信器、设备/线程亲和、进程拓扑             | 与**Wsp**                                                                                              | **Cval** + **Drv**                                      |
| **Idx**    | **索引图式**：DOF 编号、稀疏图、邻接 **结构**              | 与**Glb**：Idx **不**完成方程累加                                                                | **Redu**（消费）                                              |
| **Sync**   | **双域/双缓冲镜像**：对齐、握手、版本戳                          | 与**Wsp**：Sync **语义**对齐；Wsp **内存**复用                                             | 视域                                                                |
| **Mgr**    | **门面管理器**：聚合 Reg/Dsp、对外统一入口                       | 与**Domain**：Mgr **有**子模块引用；Domain **更薄**                                        | **Drv**                                                       |
| **Domain** | **域入口薄门面**：转发到 Ops/Mgr，几乎无算法                     | 与**Mgr**                                                                                              | **Drv**                                                       |
| **Diag**   | **诊断/探针**：计数、计时、可选断言 **不改变主方程残差**   | 与**Ops**：Diag **不**装本构主路径                                                               | 附带 /**Drv**                                                 |
| **API**    | **对外稳定 ABI**：Harness/FFI **边界封装**                 | 与**Brg**：API **稳定**承诺更强                                                                  | **Cval** + **Drv**                                      |

##### G. 互斥与择一（防重复）

- **Map ⊕ Sym**：建语义树 **Map**；只做标识符决议 **Sym**。
- **Strat ⊕ Solv ⊕ Step**：**选策略 Strat**；**解一次 Solv**；**怎么进下一步 Step**；**当前袋 Run**。
- **Reg ⊕ Dsp**：静态表 **Reg**；动态解析 **Dsp**。
- **Loc ⊕ Ops ⊕ Glb**：单角色独占才用 **Loc/Glb**；否则默认 **Ops**。
- **Asm ⊕ Glb**：有 **阶段/管线** 语义 → **Asm**；仅平坦累加 → **Glb**。
- **Brg ⊕ Pop ⊕ Wb**：跨层泛适配 **Brg**；只 Populate **Pop**；只写回 **Wb**。
- **Def ⊕ Glu**：纯声明 **Def**；有薄可执行粘合 **Glu**。
- **Proc ⊕ Orc**：SIO 形态 **Proc**；域内壳 **Orc**。
- **Proc ⊕ Drv**：**Proc** 以 **SIO 签名** 为锚；**Drv** 为 **功能驱动面**（步进/作业外壳），可与 **Proc** 分文件；二者 **不** 要求与 ProcKind 一一强制对齐。
- **Wsp ⊕ Env**：内存工作区 **Wsp**；MPI/设备 **Env**。

**目录 vs 文件名**：仓库常见 **`Types/` 目录** 下放 **`PH_Cont_Def.f90`** 这类文件——**概念上的「Types 物理分区」不强制对应 `_Types.f90` 后缀**；若团队希望字面统一，可约定「仅在纯 TYPE 桶内使用 `_Types.f90`」，但与现有 **`_Def`** 并存时需二选一或在域内统一。

**弃用文件名后缀 `_Algo`（新文件起）**：**`Algo` 默认绑定 `TYPE(..._Algo)`**，勿再作 `.f90` 尾巴。迁移时 **查 A–F 全表择一**，勿只记「Map/Strat/Reg/Asm 四字」。**存量 `*_Algo.f90` 不强制批量改名**；新拆文件与 `CONTRACT.md` 映射优先。

**选后缀速查（从旧 `*_Algo` 迁出，按专职）**：

| 若文件主要做…                                 | 新后缀优先选                                   |
| ---------------------------------------------- | ---------------------------------------------- |
| KW/卡片字段→**语义模型树**              | **Map**                                  |
| 只做**名字/符号→内部 ID**，不建树       | **Sym**                                  |
| **选**求解器/本构分支/分析路径（选项面） | **Strat**                                |
| **解一次**代数迭代/牛顿内核              | **Solv**                                 |
| **控制**时间步长、载荷增量进给           | **Step**                                 |
| 静态材料/单元/插件**注册表**             | **Reg**                                  |
| **vtable/函数指针** 运行时解析           | **Dsp**                                  |
| 全局装配**阶段机/管线**                  | **Asm**                                  |
| 平坦 `K+=Ke`、无阶段语义                     | **Glb**                                  |
| 仅单元/IP 局部算子                             | **Loc**                                  |
| 仍难归类                                       | **Ops** / **Brg** / **Proc** |

**弃用文件名后缀 `_Ctx`（新文件起）**：**`Ctx` 保留给 `TYPE(..._Ctx)` 与矩阵列**。迁移时 **查 E+F 与 G 节择一**。**存量 `*_Ctx.f90` 不强制批量改名**。

**选后缀速查（从旧 `*_Ctx` 迁出，按专职）**：

| 若文件主要装/做…                             | 新后缀优先选                                   |
| --------------------------------------------- | ---------------------------------------------- |
| scratch、缓冲池、workspace 槽位               | **Wsp**                                  |
| MPI/设备/线程拓扑、日志等级                   | **Env**                                  |
| **快照袋**（计数、阶段 ID、只读 token） | **Run**                                  |
| **步进控制流**（非策略卡片、非代数核）  | **Step**                                 |
| 仅薄 `TYPE` 合拍 + accessor                 | **Glu** 或 **Def**                 |
| 专填 L4 Populate                              | **Pop**                                  |
| 专写回结果                                    | **Wb**                                   |
| 泛跨层适配                                    | **Brg**                                  |
| 仍难归类                                      | **Ops** / **Proc** / **Orc** |

**不推荐新文件再用的历史尾巴**：带数字尾巴的历史分裂后缀仅作遗留兼容；新代码应通过子目录或更具体 basename 区分，而不是再叠数字。

**完备集的边界**：不含 **ProcKind 四后缀** 作默认文件名；不含 **`UFC/tools/`** 脚本名。拟增 **`Harness` / `Tst` / `Fmt`** 等：先评估是否与 **H**arness/**T**est/**I**O 域目录重复，再 **RFC + 表增行 + 白名单**。

### 1.1 数据四型（DataKind）

| DataKind        | 角色       | 典型内容                                 |
| --------------- | ---------- | ---------------------------------------- |
| **Desc**  | 只读配置   | 材料常数、单元截面、边界类型标识等       |
| **Ctx**   | 上下文胶水 | 步进时间、通信拓扑、工作区句柄、分支条件 |
| **State** | 运行时状态 | 应力、内变量、全局位移/残差/刚度累加目标 |
| **Algo**  | 算法描述符 | 积分阶、求解器选项、本构子模式枚举       |

**命名提醒（文件 vs TYPE）**：**`Desc` / `Ctx` / `State` / `Algo` 四字仅保留给数据四型**（对应 `TYPE(...)` 与矩阵列）。**新建 `.f90` 请勿使用 `_Desc` / `_Ctx` / `_State` / `_Algo` 后缀**。文件专职优先 **功能简称**（**`Proc` / `Drv` / `API` / `Idx`…**，见 A–F）；拿不准时 **Ops** 或 **Brg** 兜底。**ProcKind 标注**（Cval/Kern/…）与 **同名功能后缀** 若共存，在 **`CONTRACT.md` 一行消歧**（见零节）。

### 1.2 过程四型（ProcKind）

| 凝缩           | 代号        | 枚举     | 中文 | 一句话           | 典型职责                                             | 刻意不做                                 |
| -------------- | ----------- | -------- | ---- | ---------------- | ---------------------------------------------------- | ---------------------------------------- |
| **Cval** | **C** | CONTRACT | 契   | 能算吗？合法吗？ | 校验、默认值、外部输入→合法 TYPE、版本/约束断言     | 热路径大算、不写全局向量                 |
| **Kern** | **K** | KERNEL   | 核   | 给一点，还一点   | 形函数、B 阵、本构、单元 Ke/Re 等**局部** 算子 | 直接装配全局、不判收敛、不控步进         |
| **Redu** | **R** | REDUCE   | 汇   | 多点合一体       | 稀疏装配、约束装配、并行归约、索引映射               | 不判收敛、不做本构物理、不读外部输入文件 |
| **Drv**  | **D** | DRIVE    | 控   | 下一步干什么？   | 时间步、线/非线迭代、求解器路由、跨层编排            | 不直接算物理量、不直接装配               |

**执行口诀（增量步直觉）**：**先契、再核、后汇、终控**（**Cval → Kern → Redu → Drv**）。

### 1.3 理论锚点（非仓库枚举名）

P/M/F/L（Predicate、Map、Fold、Loop）可作论证锚点；工程枚举仍以 **CONTRACT / KERNEL / REDUCE / DRIVE** 为准，单字母 **C/K/R/D** 仅作表内简写。对应关系：**Cval↔断言与合法化**，**Kern↔Map（逐点映射）**，**Redu↔Fold（含稀疏结构化累加，与数学 Fold 不必逐字同一对象）**，**Drv↔Loop/Branch（含一次性路由，如静态求解器选型）**。

## 二、主责矩阵（4×4）

### 2.1 矩阵

|                            | **Desc**            | **Ctx**                      | **State**                           | **Algo** |
| -------------------------- | ------------------------- | ---------------------------------- | ----------------------------------------- | -------------- |
| **Cval**（CONTRACT） | **写**（合法 Desc） | **写**（默认 Ctx）           | 读（校验）                                | 读（校验）     |
| **Kern**（KERNEL）   | 读                        | 读                                 | **写**（局部状态：应力、SDV 等）    | 读             |
| **Redu**（REDUCE）   | —                        | 读（索引/通信）                    | **写**（全局向量/矩阵；如 K += Ke） | —             |
| **Drv**（DRIVE）     | —                        | **读+写**（路由/步进上下文） | 读（收敛等）                              | 读（算法路径） |

### 2.2 读法与边界（必读）

- **写**：该 ProcKind 对该 DataKind 的 **主要产出** 是构造或更新。
  - Kernel 写 **局部** State；Reduce 写 **全局** State（累加/装配语义）。
- **读**：执行中 **消费** 该 DataKind，但非主产出。
- **—**：**通常不由** 该 ProcKind **主责**；**不表示禁止读写**。
- **主责 ≠ 独占**：例如 Kernel 大量读 Desc；Reduce 可带少量 Algo（如积分阶）。
- **编排 vs 子步骤**：Drive 可 **触发** Contract（如自适应后重建 Desc）；主标签仍为 **D**，辅 **C**。
- **读列省略**：表中「读」只标 **典型** 路径，实际子程序可多读多型。

## 三、排除规则（防标注漂移）

用于消解「`*_Ops.f90` 全是 Kern」等现象：先看 **主数据流** 与 **写入目标**，再定主/辅标签。

### 3.1 Kern（KERNEL）

| 表象                                                | 依据            | 建议                                               |
| --------------------------------------------------- | --------------- | -------------------------------------------------- |
| 算完 Ke**直接** K_global += Ke                | 累加属 Reduce   | 主**R**，辅 **K**                      |
| 本构内超限**回退/重试**                       | 决策属 Drive    | 主**D**，辅 **K**                      |
| 读输入并 `TYPE(..._Desc)`                         | 输入→合法 TYPE | **C**                                        |
| Lamé 等预计算写入**Desc** vs **State** | 看写入落点      | 写 Desc 倾向**C**；写 State 倾向 **K** |

### 3.2 Redu（REDUCE）

| 表象                     | 依据        | 建议                               |
| ------------------------ | ----------- | ---------------------------------- |
| 装配前判定自由度是否参与 | 判定/合法化 | 主**C**，辅 **R**      |
| 装配后看 ‖R‖ 收敛      | 收敛判据    | 主**D**，辅 **R**      |
| 装配中调本构拿 Re        | 本构        | **K**（被 R 调）；R 只管累加 |

### 3.3 Cval（CONTRACT）

| 表象                   | 依据       | 建议                          |
| ---------------------- | ---------- | ----------------------------- |
| 校验后顺手预计算 Lamé | 预计算算子 | 主**C**，辅 **K** |
| 校验时打 warning       | 观测为横切 | 主**C**；日志见第四节   |

### 3.4 Drv（DRIVE）

| 表象                 | 依据                  | 建议                                        |
| -------------------- | --------------------- | ------------------------------------------- |
| 路由里调装配         | 装配                  | 主**D**（编排），**R** 为被调方 |
| 改 Ctx 子步参数      | 为分支/步进 vs 为断言 | 分支/步进→**D**；合法化→**C** |
| Drive 内直接算物理量 | 物理                  | **D** 编排，算子抽到 **K**      |

## 四、Observe（观测）政策

**问题**：日志、诊断、场输出、写文件等无处不在，若升格为独立 ProcKind，易稀释 K/R/C/D 的区分度。

**决策**：

1. **不** 将 OBSERVE 作为默认第五型 ProcKind。
2. **主分类** 仍按方程主路径标 K/R/C/D。
3. 显著观测行为：**附带** `! Obs: <说明>`。
4. **唯一目的** 为观测（如 `WriteFieldOutput`、`LogStepSummary`）：主标 **Drv（DRIVE）**（何时/如何输出属控制），并 `! Obs: primary`。

**理由**：观测是横切关注点；L1 已有 `IF_Log` 等基础设施。若未来审计/性能/ODB 成为一等架构关切，可再引入第五型或放在下文「维 3」中扩展（见附录 B）。

## 五、Module 的完整形态

```
完整 Module（功能主体 + 边界） =
  数据四型 TYPE
+ 过程四型子程序（按角色组织，不要求四型俱全）
+ MODULE（Fortran 物理单元）
+ 模块级 CONTRACT.md（对外 TYPE/PROC 白名单，避免「全 PUBLIC」）
```

**绑定关系**：

- 一型过程 **不必** 对应一格矩阵代码；矩阵是 **分类与主责** 工具。
- 一子程序可有 **主 + 辅** ProcKind（如 R+K）。
- 纯容器型 Module 可 **仅有** Contract（Init/Validate）。

**目录示例（Material）**：`MD_Mat_Types.f90`（四型 TYPE）+ `MD_Mat_Elas_Ops.f90`（Init/Eval/Assemble/Config 等映射 C/K/R/D）+ `CONTRACT.md`。

## 七、Fortran 蓝图（示例，入库前须与现有 UF_DataKind 对齐）

以下路径为 **建议落点**，实现前请在 `ufc_core` 内检索是否已有同名模块，避免重复定义。

### 7.1 ProcKind 与 DataKind

- **DataKind**：与现有 **`UF_DataKind` 或等价枚举** 对齐（`UF_DATAKIND_DESC` …）。

### 7.2 主责查询：`UF_RESP_*` 与表格「读+写」

查询接口用 **读 / 写 / 无** 三值编码矩阵。对于表中 **Drv × Ctx = 读+写**：实现里 **写作 `UF_RESP_WRITE`**（以写为主责、读为伴生），与第二节「主责≠独占」一致；若将来工具链要区分「双主责」，再扩展 RESP 枚举或位掩码。

```fortran
!===============================================================================
! UF_ProcKind_Mod — 过程角色枚举（蓝图）
! 建议：UFC/ufc_core/L1_IF/Base/UF_ProcKind.f90
!===============================================================================
MODULE UF_ProcKind_Mod
  USE IF_Prec_Algo, ONLY: i4
  IMPLICIT NONE
  PUBLIC

  ENUM, BIND(C)
    ENUMERATOR :: UF_PROCKIND_CONTRACT = 1
    ENUMERATOR :: UF_PROCKIND_KERNEL   = 2
    ENUMERATOR :: UF_PROCKIND_REDUCE   = 3
    ENUMERATOR :: UF_PROCKIND_DRIVE    = 4
  END ENUM

  INTEGER(i4), PARAMETER :: UF_PROCKIND_COUNT = 4_i4

  CHARACTER(LEN=10), PARAMETER :: UF_PROCKIND_NAMES(4) = [ &
    'CONTRACT ', 'KERNEL   ', 'REDUCE   ', 'DRIVE    '  ]

CONTAINS

  PURE FUNCTION UF_ProcKind_Name(kind) RESULT(name)
    INTEGER(i4), INTENT(IN) :: kind
    CHARACTER(LEN=10) :: name
    IF (kind >= 1 .AND. kind <= UF_PROCKIND_COUNT) THEN
      name = UF_PROCKIND_NAMES(kind)
    ELSE
      name = 'UNKNOWN  '
    END IF
  END FUNCTION UF_ProcKind_Name

END MODULE UF_ProcKind_Mod
```

!===============================================================================

!===============================================================================
MODULE UF_RespMatrix_Mod
  USE UF_ProcKind_Mod

  IMPLICIT NONE
  PRIVATE
  PUBLIC :: UF_RESP_READ, UF_RESP_WRITE, UF_RESP_NONE, UF_RespMatrix_Get

     UF_RESP_WRITE,  UF_RESP_WRITE,  UF_RESP_READ,   UF_RESP_READ,   &  ! CONTRACT

## 八、与六层、四链、SIO 的关系

| 对象                      | 回答的问题         | 与 ProcKind 关系                                                                                                |
| ------------------------- | ------------------ | --------------------------------------------------------------------------------------------------------------- |
| **Layer（L1–L6）** | 在哪里、依赖朝哪边 | **正交**：`定位 ≈ Layer + ProcKind + DataProfile`；勿把层数与角色数加成一个「维度」                    |

**触发条件示例**：审计级日志、性能诊断一等公民、ODB 写入与方程主路径 **同等** 架构治理。
