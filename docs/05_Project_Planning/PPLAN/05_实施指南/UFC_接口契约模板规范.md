# UFC 接口契约模板规范

## 1. 文档目的

本文定义 UFC 中层架构中的 **接口契约（Interface Contract）** 模板规范，用于统一描述模块、域、层之间稳定交互面的：

- 接口语义
- 参数模型
- 输入输出约束
- 生命周期
- 热路径要求
- 错误出站
- 时序与兼容策略
- 测试与守卫追溯映射

本文档与 `UFC_域级合同卡模板规范.md` 配套使用：

- 域级合同卡回答：**这个域是什么**
- 接口契约回答：**这个域如何与外部交互**

---

## 2. 适用范围

本规范适用于以下接口类型：

1. **跨层接口**
   - L3 → L4
   - L4 → L5
   - L6 → L3/L5
   - Bridge / Populate / WriteBack 接口

2. **跨域接口**
   - Material ↔ Section
   - Element ↔ Material Driver
   - Contact ↔ Assembly
   - Constraint ↔ Solver

3. **共享域核心接口**
   - Init / Build / Register
   - Compute / Apply / Update
   - Bind / Populate
   - Export / WriteBack

4. **高复杂度接口**
   - 使用 `Arg / Desc / State / Algo / Ctx`
   - 位于热路径
   - 具有状态更新语义
   - 具有用户子程序映射需求
   - 存在多实现替换关系

---

## 3. 接口契约的定位

接口契约位于以下内容之间：

- 上承：域级合同卡、层级建模文档、总纲规则
- 下接：子程序签名、实现代码、测试用例、guardian / harness 绑定

接口契约必须明确：

1. 这个接口完成什么；
2. 这个接口不应该完成什么；
3. 调用方必须准备什么；
4. 被调方承诺返回什么；
5. 调用期间哪些对象可读、可写、可回写；
6. 热路径与错误路径如何受控；
7. 版本与兼容如何演化。

---

## 4. 编写原则

### 4.1 原则一：语义先于签名
先说明接口做什么，再说明参数长什么样；禁止只列参数不写语义。

### 4.2 原则二：对象成组传递
优先采用对象槽位表达，而不是扁平参数堆叠：

- `Arg`
- `Desc`
- `State`
- `Algo`
- `Ctx`
- `Result`
- `Status/Error`

### 4.3 原则三：调用边界清晰
要明确：
- 调用前置条件
- 调用期间可修改对象
- 调用后承诺结果
- 失败后的状态保证

### 4.4 原则四：热路径可控
若接口位于热路径，必须显式写：
- 禁止事项
- 可缓存项
- 可复用项
- 不允许的动态行为

### 4.5 原则五：可测试可追溯
接口契约不能只是文字说明，必须能映射到：
- guardian 规则
- harness 入口
- 正负路径测试
- 兼容回归点

## 5. 标准模板结构

以下为 UFC 接口契约建议标准结构。

---

## 5.A 基本信息

- 接口名：
- 所属层级：
- 所属域 / 子域：
- 接口类型：跨层 / 跨域 / 热路径 / 用户扩展 / Bridge / Populate / WriteBack
- 版本：
- 状态：Draft / Active / Stable / Deprecated
- 对应代码路径：
- 对应域级合同卡：

### 编写要求
- 接口名应尽量与实际公开过程或逻辑入口一致。
- 若是逻辑接口而非单一过程，也应标明主要落点过程族。

---

## 5.B 语义目标

### 本接口负责
- 用 2～5 条说明该接口完成的语义目标

### 本接口不负责
- 用 2～5 条说明禁止越界内容

### 调用结果承诺
- 成功调用后保证什么
- 失败调用后保证什么

### 编写要求
- 必须能与相邻接口区别开来。
- 不得使用“处理相关逻辑”等模糊表述。

---

## 5.C 调用角色与方向

- 调用方：
- 被调方：
- 调用方向：
- 上游准备对象：
- 下游消费对象：
- 是否允许链式调用：

### 编写要求
- 对于跨层接口，应明确层间方向。
- 对于共享域接口，应说明与三层协同表的对应关系。

---

## 5.D 参数模型

建议统一按以下槽位描述：

- `Arg`
- `Desc`
- `State`
- `Algo`
- `Ctx`
- `Result`
- `Status/Error`

### 每个槽位需说明
- 是否必需
- 是否只读
- 是否可更新
- 是否可为空
- 生命周期
- 所有权
- 典型字段

### 编写要求
- 优先避免扁平参数风格。
- 若历史接口仍为扁平参数，应给出映射表。

---

## 5.E 数据约束

需显式说明：

- 单位体系
- 维度/shape
- index 规则
- 枚举/标签合法值
- 默认值
- 空值语义
- alias / pointer 约束
- 排序/去重要求（若有）

### 编写要求
- 数据约束必须可供后续测试/静态检查使用。
- 对关键字段建议写出“非法示例”。

---

## 5.F 生命周期与所有权

明确：

- 谁初始化
- 谁持有
- 谁更新
- 谁释放
- 是否可跨 step / inc / iter 存活
- 是否可缓存
- 是否允许复用 scratch

### 编写要求
- 必须明确 `State` 与 `Ctx` 的所有权差别。
- 若接口失败，需说明对象是否回滚。

---

## 5.G 时序契约

需要说明：

- 前置条件
- 后置条件
- 允许调用时机
- 禁止调用时机
- 是否要求先 Register / Build / Populate / Bind
- 是否允许重复调用
- 是否允许并行调用

### 编写要求
- 这部分是后续 harness 最容易转化为前置测试条件的内容。
- 必须避免“默认总是可调用”。

---

## 5.H 热路径契约

若接口位于热路径，必须额外说明：

- 热路径入口
- 典型循环粒度
- 禁止事项
- 允许事项
- 缓存策略
- 是否允许日志
- 是否允许全局查询
- 是否允许字符串解析
- 是否允许动态分配

### 编写要求
- 非热路径接口也可写“本接口不处于热路径”。
- 热路径约束必须结合真实执行面，不得模板化空写。

---

## 5.I 错误契约

需说明：

- 错误码/状态对象
- 错误等级
- 可恢复/不可恢复分类
- 失败后状态保证
- 是否允许降级
- 必须回传的诊断锚点

### 建议锚点
- layer
- domain
- subroutine / interface name
- object id / entity id
- step / increment / iteration（如适用）

### 编写要求
- 若接口允许部分成功/部分失败，应明确其语义。
- 错误契约必须与实现错误出站方式一致。

---

## 5.J 兼容与演化策略

说明：

- 新字段如何增加
- 旧字段如何弃用
- 旧接口如何保留兼容层
- 何时允许 breaking change
- 如何标识接口版本切换

### 编写要求
- 该部分用于避免未来模板和实现漂移。
- 若当前无版本机制，也应先预留占位。

---

## 5.K 治理与验证映射

建议至少说明：

- 对应域级合同卡
- 对应 guardian 规则
- 对应 harness profile / alias
- 最小正路径测试
- 最小负路径测试
- 热路径回归点
- 性能/稳定性风险点

### 编写要求
- 使接口契约能进入“文档—守卫—测试”闭环。

---

## 7. 评审检查清单

编写或评审接口契约时，建议检查：

1. 是否先写清语义，再写参数；
2. 是否清楚区分 `Arg / Desc / State / Algo / Ctx`；
3. 是否明确哪些对象可读、可写、可回写；
4. 是否明确失败后的状态保证；
5. 是否明确热路径限制；
6. 是否明确调用前置条件与调用时机；
7. 是否明确错误锚点与可恢复性；
8. 是否为后续版本演化预留策略；
9. 是否能映射到 guardian / harness；
10. 是否能直接为模板子程序和测试用例提供输入。

## 9. 结论

接口契约模板规范的目的，是把 UFC 从“靠过程名和历史习惯理解调用关系”推进到：

**靠显式语义、参数槽位、时序边界、错误出站和验证映射来稳定接口演化。**

后续若 UFC 要稳定扩展共享域、模板实现和用户子程序接口，对关键交互面建立统一接口契约是必要前提。
