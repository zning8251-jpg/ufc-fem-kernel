## 1. 文档目的

沉淀以下结论，供裁剪、重组与门禁对齐使用：

- 约定 **MVP 分析范围**（与步类型、几何非线性、接触、输入路径相关）。
- 澄清 **L3 是否「域级完整即不必改」**。
- 给出 **L4_PH、L5_RT 一级域清单**（与仓库文件夹一致），及 **MVP 下各域优先级**（必留 / 按需 / 可延后裁剪）。
- 与 **四链**、**HYPLAS 淬炼**（L3 只读 Desc、Populate、热路径约束）及 **三层集成整改** 衔接。

## 2. MVP 定义（当前共识）

| 项 | 范围 |
|----|------|
| **分析类型** | **结构**：**隐式静力**、**隐式动力**、**显式动力** |
| **非线性** | **几何非线性（NLGEOM）** 纳入 MVP |
| **接触** | **接触**纳入 MVP |
| **其它分析步** | 热、土力学、声、耦合场等 **暂不实现**（步类型枚举与同步路径应显式标为不支持或 Phase 2+） |
| **L6 输入** | 若可 **单独后期插入**，MVP 阶段 **可先拿掉**；L3 需能通过 **程序化组模 / 最小装配 API** 填满 Desc |

**验收建议**：至少一条可自动化或手动的 **端到端用例**（静力 + 一种动力 + 含接触 + NLGEOM 开关），再扩展矩阵。

---

## 3. L3_MD：是否「域级完整、不需要改」？

**结论：不是。** 区分两件事：

1. **域级分类**  
   对「结构 + 三类动力学步 + NLGEOM + 接触」，L3 侧通常仍需：**Mesh、Material、Section、Assembly/Part、Step、LoadBC、Interaction/Constraint（接触对/表面）、DOF** 等概念域。这说的是 **概念覆盖**，不是「实现已齐、目录冻结」。

2. **MVP 必须在 L3 推进的工作**  
   - **Step / 分析类型**：仅保留隐式静力、隐式动力、显式动力；其它步在 Desc/枚举/同步中 **明确延后**。  
   - **NLGEOM**：与 Step/Model 描述字段及 L4/L5 契约对齐（标志位、增量策略约定等）。  
   - **接触**：Interaction/Constraint 与表面/对的最小 **稳定数据模型**，与 L4 接触本构、L5 搜索/求解接口 **合同一致**。  
   - **无 L6 时期**：补齐或收紧 **非 KW** 数据注入路径；可删减或 stub 掉 **仅输入层** 才需要的复杂分支（如部分 Output/Amplitude 可先最小子集）。

因此：**L3 不必大改「域集合」本身，但要为 MVP 做范围冻结、合同与同步路径收敛**；与「完整不用改」不是同一命题。

---

## 4. L3 / L4 / L5 域集合是否要一一镜像？

**否。** 一致性重在：

- **数据链**：Desc/Ctx/State 经 Bridge/Populate 可追溯；  
- **四链**：理论 / 逻辑 / 计算 / 数据在文档与接口上可对照；  
- **依赖方向**：遵守六层与 Guardian（如 L3 不反向依赖 L4/L5 模块名，Bridge 路径按规范）。

L3 域集通常 **最全**；L4 以 **物理算子** 为主裁剪；L5 以 **调度与运行时** 为主裁剪。**不要求** 三层文件夹同名同数量。

## 5. L4_PH 一级域清单（与 `ufc_core/L4_PH/` 一致）

| 域（一级目录） | 职责摘要 |
|----------------|----------|
| **Bridge** | L3→L4 Populate、几何与模型数据下传（如 `PH_Brg_L3`、`PH_L4_Populate_Core` 等） |
| **Constraint** | 与约束/接触表述相关的物理侧类型与算法片段（视实现与接触模型而定） |
| **Contact** | 接触本构、接触域类型、BVH/几何支撑等 **接触力学核** |
| **Coupling** | 多物理耦合相关物理核（纯结构 MVP 可整域 **延后**） |
| **Element** | 单元族与单元核（SOLID/SHELL/BEAM 等子目录） |
| **LoadBC** | 与边界/载荷在物理离散或映射相关的类型与转换（如 flat/nested） |
| **Material** | 本构、材料状态、注册、UMAT 桥等（ELA/HYP/PLG/… 子目录） |

**说明**：`L4_PH` 根目录下另有 **单文件**（如 `PH_L4_Populate_Core.f90`），逻辑归属 **Populate/Bridge**，不单独算作第八个文件夹域。

## 6. L5_RT 一级域清单（与 `ufc_core/L5_RT/` 一致）

| 域（一级目录） | 职责摘要 |
|----------------|----------|
| **Assembly** | 装配、刚度/内力集成、NL 材料/几何求值与求解衔接 |
| **Bridge** | L5↔L4/L3 运行时桥（如材料、单元侧） |
| **Contact** | 接触 **运行时**：表面、搜索、接触求解/控制、与步进衔接 |
| **Coupling** | 耦合步进、多物理 Runner（纯结构 MVP 可 **延后**） |
| **Element** | 运行时单元调度、UMAT/形函数链等 |
| **LoadBC** | 载荷/BC **施加**与步内更新、相关上下文 |
| **Logging** | 日志与诊断（宜保持薄实现） |
| **Material** | 运行时材料求值调度 |
| **Mesh** | 运行时网格/连续体网格胶水类型（如与 Bridge 组模衔接） |
| **Output** | 场输出与写控制 |
| **Solver** | 对接 L2、非线性迭代、时间积分等 **求解外壳** |
| **StepDriver** | **步进总控**：静力、隐式动力、显式动力等 Runner、增量/迭代/Job |
| **UEL** | 用户单元扩展运行时（无 UEL 需求可 **延后**） |
| **WriteBack** | 解回写与审计/白名单（建议 MVP 至少保留最小通路） |

## 7. MVP 与 L4/L5 域对照（实施优先级）

### 7.1 L4_PH

| 域 | MVP 建议 |
|----|----------|
| Bridge | **必**（Populate） |
| Element | **必**（先收敛单元族子集） |
| Material | **必**（先收敛本构子集；大变形与 NLGEOM 与 HYP 等强相关） |
| Contact | **必** |
| LoadBC | **按需**（取决于边界积分与 L3 载荷格式映射是否走 L4） |
| Constraint | **按需**（与接触/约束统一建模时保留最小） |
| Coupling | **可整域延后** |

### 7.2 L5_RT

| 域 | MVP 建议 |
|----|----------|
| StepDriver | **必**（`Static` + `DynImpl` + `DynExpl` 子集；其它 Runner 不纳入 MVP） |
| Assembly | **必** |
| Solver | **必** |
| Contact | **必** |
| LoadBC | **必** |
| Material / Element | **必**（运行时调度，常与 Assembly/StepDriver 紧耦合） |
| WriteBack | **强烈建议保留**（可先最小实现） |
| Output | **按需**（可先最简） |
| Mesh | **按需**（取决于热路径快照与组模策略） |
| Bridge | **按需**（保留现有最小依赖集） |
| Logging | **薄保留** |
| Coupling / UEL | **MVP 可裁** |

## 9. 按需域实施安排（L4 LoadBC/Constraint；L5 Output/Mesh/Bridge/Logging）

本节回答 MVP 文档 §7 中「按需」域 **如何落地**：保留什么职责、放在哪一层、能否删目录。

### 9.1 L4：`LoadBC`（按需）

**安排原则**：L4 的 LoadBC **只保留「物理离散或核内所需」** 的部分，不把「步内施加、方程号映射」放在 L4。

| 情形 | 建议 |
|------|------|
| **边界积分**（面力、压力、沿边分布载荷需在单元/面上用形函数积分进等效节点力） | 在 **Element**（或面单元/接触面共用核）内实现 **积分核**；L4 `LoadBC` 仅保留 **输入侧类型/缓冲区约定**（如 facet 载荷与局部坐标），避免与 Element 重复两套 N^T·t 逻辑。 |
| **格式转换**（L3 嵌套 LoadBC ↔ L4/ L5 所需的 flat 缓冲、与 Populate 对齐） | 保留 **薄模块**：转换子程序 + 共享类型，供 **Populate（Bridge）** 或 **L5 LoadBC 施加前** 调用；若转换已集中在 `PH_L4_Populate_*` 且不再复用，可 **合并进 Bridge 单文件**，但需在合同中标出「转换责任在 Populate」。 |
| **MVP 仅集中力/已简化为方程级载荷**（无 facet 分布载荷） | L4 `LoadBC` 可 **仅类型 + no-op 或删除未引用实现**；载荷物理量由 L5 写入右端项。 |

**结论**：按需 = **先画清「谁在积分、谁在转换」**；积分归 Element/Contact 边，转换归 Populate 或薄 LoadBC，**不要为了保留文件夹而复制逻辑**。

### 9.2 L4：`Constraint`（按需）

**安排原则**：与 **Contact** 强重叠时，**以 Contact 为力学主域**，Constraint 不并行堆一套求解接口。

| 情形 | 建议 |
|------|------|
| **接触 + 粘结/绑定** 与节点 MPC 统一用「约束单元」或同一套乘子/罚函数框架 | **Contact（+ Element）** 承载核心；`Constraint` 保留 **共用 Desc/状态片段**（若有）或 **迁入 Contact** 下 `Shared`，目录可 **缩成 1～2 个类型文件**。 |
| **仅位移 BC**（本质代数约束） | 主要在 **L3 Desc + L5 施加**；L4 **无独立 Constraint 核** 亦可，除非单元公式需要「罚弹簧 BC」等物理近似。 |

**结论**：按需 = **最小共用类型 + 单一主域（通常 Contact）**；避免 L4 Constraint 与 L4 Contact 各维护一套约束残差。

### 9.3 L5：`Output`（按需；是否散入各子域？）

**不建议**把 IO 与格式细节 **散进** Assembly、Contact、Element 等各域。

**推荐**：保留 **集中、薄的 `Output` 域**（门面）：

- **职责**：输出请求解析（或程序化列表）、打开/关闭文件、步频与字段注册；**不写**具体单元应力积分。
- **数据获取**：从 **已约定的 Ctx/State**（或 DataPlatform）**拉取** 节点位移、单元应力等；各算子域只负责 **填满合同字段**，不直接 `WRITE`。

**最简 MVP**：一种格式（如 CSV/VTK 之一）+ 少量字段 + 在 StepDriver 末尾或固定增量回调 **单入口** 调用。

### 9.4 L5：`Mesh`（按需；是否热路径快照？）

与 **HYPLAS 淬炼** 及现有整改方向一致：**推荐热路径使用「只读网格视图」**（如 `RT_MeshSnapshot` 或与 `MD_Mesh_*_Arg` 等价的 **注入快照**），避免在单元/装配循环中 **`USE` 完整 L3 `MD_Mesh_Domain`**。

| 角色 | 建议 |
|------|------|
| **步内循环 / 接触搜索 / 单元积分** | **快照或等价 Arg 包**（坐标、连接、截面索引等），在步前或重网格事件后 **重建**。 |
| **组模 / 注册 DataPlatform** | 仍可从 L3 或 Bridge **一次性**构建；可继续用 **胶水模块**（如 `ContMeshCtx` 类）放在 `L5_RT/Mesh` 或并入 Bridge，**不必**在热路径保留「胖」网格对象。 |

**结论**：**最有方案 = 热路径只读快照（或等价注入）**；`L5_RT/Mesh` 可 **极薄**（仅胶水类型 + Build），与「整域删除」不同——删除的是 **热路径对 L3 胖类型的依赖**，不是「不需要网格数据」。

### 9.5 L5：`Bridge`（按需；能否移除？）

**不建议整域移除**，除非把跨层过程 **全部** 迁到单一模块（往往造成 StepDriver/Assembly **膨胀**、依赖更难查）。

**「最小集合」含义**：

- 保留 **少量模块**，专门做 **L5↔L4 或 L5↔L3 的边界调用**（如材料点调度、Populate 回调），满足 Guardian 与 **单向依赖**。
- **可以做的**：合并文件、改名、删除 **从未被链接** 的桥文件；**不建议**：零桥接层导致 L5 直接散落 `USE MD_*` 违反分层。

### 9.6 L5：`Logging`（薄保留 vs 暂时移除）

| 选项 | 说明 |
|------|------|
| **薄保留（推荐）** | 统一走 **L1 `IF_Log` / 错误 API**；L5 仅 **薄包装或零成本宏**；关闭日志时无热路径开销。 |
| **暂时移除 L5 域目录** | 仅当 **全仓库无 `USE` L5 专用 Log 模块** 且调用点已改为 L1；否则删目录会 **机械破坏编译**。若「无用」= 无业务日志需求，可 **保留空壳或 stub 模块** 占位，不必删文件夹。 |

**结论**：优先 **薄门面 + 关断**；**不要为了「干净」删目录** 除非已完成全量替换与构建验证。
