## 一、核心方法论

### 1.1 三原则

| 原则 | 含义 | 验证手段 |
|------|------|---------|
| **数据单源** | L3_MD 是唯一真相源；L4/L5 通过 Populate 提取，热路径不回读 L3 | 检查每个 [IN] 的最终溯源是否到达 L3 或外部输入 |
| **目标倒推** | 从最终交付物（stress, Ke, 收敛判断…）倒推需要的数据，递归到 L3 真相 | 倒推树无断裂 |
| **生产/消费闭合** | 每份数据有且仅有一个生产者，至少一个消费者；无悬空 | 闭合性矩阵全打勾 |

### 1.2 推导流程

```
Step A: 确定最终目标（本域对外交付什么）
         ↓
Step B: 倒推目标所需的每份数据
         ↓
Step C: 对每份数据递归追溯生产者
         ↓ （直到触达 L3 真相源 / 外部输入 / 前序步生产）
Step D: 正向排列步骤（拓扑排序：生产者在消费者之前）
         ↓
Step E: 为每步填写五要素卡
         ↓
Step F: 闭合性验证（无悬空生产、无无源消费、无环）
```

### 1.3 域类型与推导策略

| 域类型 | 目标倒推起点 | 典型步骤模式 |
|--------|------------|------------|
| **计算域** | 物理量（stress, Ke, gap…） | 公式分解 → 中间量链 → 输入参数 |
| **编排域** | 状态转移完成 | 状态机拆解 → 转移条件 → 子调用 |
| **数据域** | 数据结构完备 | CRUD 生命周期 → 校验 → 索引 |
| **桥接域** | 下游 slot 填满 | 源字段 → 映射 → 目标字段 |
| **观测域** | 日志/计时输出 | 事件触发 → 格式化 → 写出 |

## 二、算法步卡格式（五要素）

每个算法步使用以下标准格式：

```
### Step {N}: {过程名片段} — {一句话意图}

**设计意图**: {WHY — 这一步为什么必须存在，物理/逻辑理由}

**消费 [IN]**:
| 数据 | 来源 TYPE.field | 生产者 | 温度 |
|------|----------------|--------|------|
| {名称} | {Desc.E / ctx.D_el / 外部形参} | {Step N-x / Populate / 外部} | {冷/温/热} |

**生产 [OUT]**:
| 数据 | 目标 TYPE.field | 消费者 | 温度 |
|------|----------------|--------|------|
| {名称} | {Desc.G / ctx.D_el / 形参 OUT} | {Step N+y / 外部 Element} | {冷/温/热} |

**算法核**:
```text
{数学公式 / 逻辑伪代码 / 状态转移}
```

**前置条件**: {执行前必须成立的断言}
**后置保证**: {执行后保证成立的断言}
**Phase**: {Config/Populate/Step/Increment/Iteration/Local}
**复杂度**: {O(?)}
```

## 三、闭合性验证矩阵

算法步规约完成后，填写闭合性验证矩阵：

```
| 数据项 | 生产者(Step) | 消费者(Step) | 闭合? |
|--------|-------------|-------------|-------|
| desc.E | Step 0 (Populate) | Step 1, 2 | ✓ |
| desc.G | Step 2 | Step 3 | ✓ |
| ctx.D_el | Step 3 | Step 4, 5 | ✓ |
| stress(6) | Step 4 | 外部(Element) | ✓ |
| strain(6) | 外部(Element) | Step 4 | ✓ (外部输入) |
```

闭合条件：
- 每行"生产者"列非空 → 无无源消费
- 每行"消费者"列非空 → 无悬空生产
- 无循环依赖 → 可拓扑排序

## 四、跨域数据流标注

当算法步涉及跨域数据时，使用以下标注：

```
消费 [IN]: strain(6)  ← 【跨域】L4_PH/Element.Ctx → B*u 计算产出
生产 [OUT]: stress(6) → 【跨域】L4_PH/Element.IP_loop → Fint 积分消费
```

跨域数据流在闭合性验证中标注为"外部(域名.过程)"，在全栈跨域图中统一追踪。

## 五、使用指南

1. **新域开发**：先写推演卡（CONTRACT→过程清单），再写算法步规约（每过程→五要素步）
2. **已有域审查**：读代码 → 逆向提取算法步 → 验证闭合 → 发现缺失/冗余
3. **代码生成**：每个 Step 直接对应一个 SUBROUTINE 或 SUBROUTINE 内的一段连续逻辑
4. **测试生成**：前置/后置条件直接转化为单元测试断言
