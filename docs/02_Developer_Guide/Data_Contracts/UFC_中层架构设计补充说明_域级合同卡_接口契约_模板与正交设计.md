# UFC 中层架构设计补充说明：域级合同卡、接口契约、模板与正交设计

## 1. 文档目的

本文档用于沉淀以下讨论成果，作为后续制定任务、补充总纲、拆分实施路线与约束代码治理的参考底稿：

1. 是否需要“层级-域级（子域）合同卡”，其定义、用途与建议结构；
2. “接口契约”与“域级合同卡”的区别与配套关系；
3. L3 / L4 / L5 共享/重叠域是否应同步设计，以及如何设计；
4. 模板子程序、模板 TYPE、用户子程序接口对齐的必要性与落地方法；
5. 分析类型与 Element / Section / Material / Procedure 正交设计，如何避免组合爆炸；
6. 在上述五点之外，UFC 架构还应补充关注的关键设计问题。

本文不替代：
- `docs/05_Project_Planning/PPLAN/01_架构总纲/UFC_架构设计总纲_深度整合版_v5.0.md`
- 各域详细设计文档

## 2. 总体判断：UFC 当前最需要补的是“中层架构件”

UFC 现阶段已经具备较强的总纲表达能力，但若要进一步支撑：

- 多域并行扩展；
- 54 类材料族与 245 类单元族的持续扩库；
- L3/L4/L5 共享域协同设计；
- 用户子程序式扩展；
- guardian / harness / 文档三位一体治理；

则不能只依赖总纲和代码实现，还需要补出一层稳定的 **中层架构件**。

建议明确 UFC 中层架构件至少包括以下 5 类：

1. **域级合同卡（Domain Contract Card）**
2. **接口契约（Interface Contract）**
3. **三层协同表（L3/L4/L5 Shared-Domain Collaboration Table）**
4. **模板子程序 / 模板 TYPE / 模板文档**
5. **兼容矩阵 / 能力标签表（Compatibility Matrix / Capability Tags）**

可将 UFC 架构文档体系理解为：

- **总纲层**：原则、边界、规则、治理
- **中层架构件层**：域说明、交互合同、模板、组合规则
- **设计实例层**：某一材料域/单元域/接触域的具体落地
- **代码实现层**：Fortran 模块、过程、测试、守卫、harness

## 3. 域级合同卡：是什么、为什么需要、应该包含什么

### 3.1 定义

**域级合同卡（Domain Contract Card）**  
是某个域或子域在 UFC 六层架构中的最小设计说明单元，用于明确：

- 该域是什么；
- 该域负责什么、不负责什么；
- 该域有哪些核心对象；
- 该域的数据如何流动；
- 该域的热路径边界在哪里；
- 该域暴露哪些子程序族与扩展点；
- 该域如何被 guardian / harness / 文档约束。

它位于“架构总纲”与“详细设计/代码实现”之间。

### 3.2 作用

域级合同卡主要解决以下问题：

1. **防止同名域语义漂移**  
   例如 Material、Element、LoadBC、Contact 在 L3/L4/L5 各层均可能存在，如果没有域级合同卡，各层会逐渐形成不同语义。

2. **防止子域职责膨胀**  
   例如 Populate、Bridge、WriteBack 等模块逐步变成“万能层”。

3. **防止四类 TYPE 漫无边界地扩张**
   - `Desc`
   - `State`
   - `Algo`
   - `Ctx`

4. **防止热路径被冷路径逻辑污染**
   - allocate / deallocate
   - 日志/字符串解析
   - 全局查询
   - 条件分发过重

5. **防止子程序族接口各写各的**
   例如某些域用 `arg + ctx`，某些域又直接展开十几个参数。

6. **为后续大规模扩库提供稳定边界**
   使新增材料、单元、载荷边界、接触等不再依赖“看旧代码猜”。

### 3.3 建议结构

建议每个核心共享域至少有一张合同卡，合同卡应包含：

#### A. 基本信息
- 域名
- 层级
- 父域 / 子域
- 状态（Draft / Active / Stable / Deprecated）
- 一句话职责

#### B. 职责边界
- 本域负责什么
- 本域不负责什么
- 与前后域的边界说明

#### C. 四类 TYPE 映射
- `Desc`
- `State`
- `Algo`
- `Ctx`

说明：
- 是否必需
- 生命周期
- 所有权
- 是否允许跨层传递

#### D. 四链映射
映射本域在以下链中的位置：
- 理论链
- 逻辑链
- 计算链
- 数据链

#### E. 计算流程图
建议以固定流程描述：
- Init
- Register / Build
- Populate / Bind
- Evaluate / Compute
- Assemble / Apply
- UpdateState / WriteBack
- Finalize

#### F. 数据结构组织
需要显式回答：
- 结构体嵌套索引如何组织
- 扁平存储如何组织
- 是否采用 registry / slot / idx 访问
- AoS / SoA / Hybrid 方案
- 冷热数据是否分层

#### G. 子程序清单
本域建议显式列出必要过程族：
- `Init`
- `Register`
- `Validate`
- `Build`
- `Populate`
- `Bind`
- `Compute`
- `Update`
- `Apply`
- `WriteBack`
- `Finalize`

#### H. 接口形态
- 是否统一采用结构体/类传递
- `Arg / Desc / State / Algo / Ctx / Result / Status` 如何组织
- 是否允许 `_Idx` 形式
- 是否有 `Bridge` 变体

#### I. 数据流转
- 上游来源
- 下游消费者
- 何处只读
- 何处可写
- 回写白名单
- 失败回滚语义

#### J. 热路径边界
- 热路径入口是什么
- 热路径禁止什么
- 热路径允许什么
- 哪些字段必须提前 Populate

#### K. 扩展点
- 模板子程序
- 注册点
- Bridge 适配点
- 用户子程序映射点
- AI 插槽

#### L. 治理与验证
- 对应 guardian 规则
- harness 入口
- 负路径测试关注点
- 文档锚点

### 3.4 结论

域级合同卡不是可选增强项，而应成为 UFC 下一阶段中层设计的基础制度件。  
建议 Material / Element / Section / LoadBC / Constraint / Contact / Solver / WriteBack / Bridge / AI / Output / Step / Procedure 等共享域均逐步配齐。

## 4. 接口契约：它是什么，与域级合同卡有什么不同

### 4.1 定义

**接口契约（Interface Contract）**  
用于规定模块、域或层之间的稳定交互方式，重点回答：

- 接口做什么；
- 不做什么；
- 如何调用；
- 输入输出是什么；
- 调用前后各自承担什么责任；
- 错误如何出站；
- 生命周期与兼容性如何约束。

### 4.2 与域级合同卡的区别

| 维度 | 域级合同卡 | 接口契约 |
|------|------------|----------|
| 关注点 | 域是什么 | 接口怎么交互 |
| 粒度 | 域/子域 | 过程/接口/调用面 |
| 目标 | 澄清职责和边界 | 稳定交互和调用规则 |
| 主要内容 | 对象、数据流、热路径、过程族 | 参数、语义、前后置条件、错误、版本 |

简化理解：
- **域级合同卡**：部门职责说明
- **接口契约**：部门之间的办事流程约定

### 4.3 是否每个域都需要独立接口契约文档

建议采用“核心域必有合同卡，复杂接口按需独立契约”的策略：

#### 必须有域级合同卡的情况
- 核心共享域
- 跨层存在的域
- 扩展频繁的域
- 未来要模板化的域

#### 建议独立成接口契约的情况
- 存在跨层调用
- 存在跨域调用
- 参数复杂
- 使用 `Arg/Ctx/Desc/State`
- 有热路径要求
- 有状态更新
- 有用户子程序映射
- 有 Bridge / Populate / AI / WriteBack 等中介行为

### 4.4 接口契约建议结构

#### A. 基础信息
- 接口名
- 所属层
- 所属域 / 子域
- 版本
- 状态

#### B. 语义目标
- 做什么
- 不做什么
- 成功返回后保证什么

#### C. 参数模型
建议统一纳入以下槽位：
- `Arg`
- `Desc`
- `State`
- `Algo`
- `Ctx`
- `Result`
- `Status/Error`

#### D. 数据约束
- 单位
- 维度
- index 规则
- 默认值
- nullable 约束
- alias / pointer 规则

#### E. 生命周期
- 谁初始化
- 谁持有
- 谁更新
- 谁释放
- 是否可跨 step / inc / iter 保留

#### F. 热路径约束
- 是否在热路径调用
- 是否允许 allocate / deallocate
- 是否允许 IO
- 是否允许访问全局容器
- 是否允许字符串解析

#### G. 时序契约
- 前置条件
- 后置条件
- 是否可重入
- 是否可并行
- 是否要求先 Populate / Bind / Register

#### H. 错误契约
- 错误码
- 降级策略
- 是否可恢复
- 是否要求携带上下文锚点

#### I. 兼容策略
- 字段新增策略
- 旧字段弃用策略
- 破坏性变更迁移策略

### 4.5 结论

域级合同卡与接口契约应配套使用：

- 域级合同卡回答“域是什么”
- 接口契约回答“域如何交互”

UFC 不宜只写总纲，也不宜只写实现，而应建立“域卡 + 契约”的中层治理体系。

## 5. L3 / L4 / L5 共享域：必须同步设计，但要“同域异责”

### 5.1 结论先行

Material / Element / LoadBC / Contact / Constraint / Solver / WriteBack / Bridge / AI 等共享域，在 L3 / L4 / L5 三层中通常都存在镜像或投影。  
这些共享域必须同步设计，但不能三层复制同一份职责。

正确原则是：

> **同域可跨层存在，但必须同域异责、合同一致、数据单向。**

### 5.2 为什么必须同步设计

若共享域不做同步设计，会出现：

1. 字段名不一致
2. 生命周期不一致
3. 状态所有权不一致
4. 回写边界不一致
5. 类型与接口演化各自为政

最终导致：
- L3 认为是定义态
- L4 认为是工作态
- L5 又把它当成运行态或回写态

### 5.3 建议引入“三层协同表”

每个共享域建议至少配一张三层协同表，示意如下。

#### 示例：Material 共享域

| 层 | 域对象 | 职责 | 输入 | 输出 | 禁止事项 |
|----|--------|------|------|------|----------|
| L3_MD | Material Domain | 材料定义、族别、参数组织、兼容标签 | keyword / section binding | material desc / registry | 不做本构积分 |
| L4_PH | Material Driver | 本构积分、切线、状态更新 | desc + state + algo + ctx | stress / tangent / new_state | 不管理全局生命周期 |
| L5_RT | Material Runtime | 步增量调度、回写、提交与回滚 | runtime context | committed state / rollback state | 不定义材料参数语义 |

#### 示例：Element 共享域

| 层 | 域对象 | 职责 | 输入 | 输出 | 禁止事项 |
|----|--------|------|------|------|----------|
| L3_MD | Element Definition | 类型、拓扑、DOF 模式、绑定关系 | mesh / section / material refs | elem desc | 不做刚度/残差计算 |
| L4_PH | Element Kernel | 形函数、积分、局部残差刚度 | desc + section + material driver | local residual / tangent | 不做全局装配 |
| L5_RT | Element Runtime | 遍历、调度、装配、并行分块 | active set / workset | assembly contributions | 不改单元数学公式 |

### 5.4 建议写成 UFC 铁律

#### 铁律 1：语义主词唯一
同一个核心概念只能有一个主定义来源，例如材料参数主定义应位于 L3。

#### 铁律 2：状态主控唯一
状态对象的创建、提交、回滚、归档必须有唯一主控层，不能三层都写。

#### 铁律 3：合同对象一致
`Desc / State / Algo / Ctx` 的字段命名与语义应尽量统一，不允许每层演化出相似但不兼容的结构。

#### 铁律 4：数据单向
定义态通常向下流，运行态仅按白名单回写。禁止低层反向定义高层语义。

### 5.5 结论

L3 / L4 / L5 并不是孤立分层，而应围绕共享域进行同步建模。  
建议后续文档化时，为 Material / Element / Section / LoadBC / Constraint / Contact / Solver / Bridge / WriteBack / AI 等全部建立共享域清单与三层协同表。

## 6. 模板子程序 / 模板 TYPE / 用户子程序接口：必须体系化

### 6.1 结论先行

如果 UFC 未来要持续扩展：
- 材料族
- 单元族
- 载荷边界
- 接触
- 约束
- AI / Bridge
- 用户扩展点

则必须提供模板体系，否则所有新增实现都将退化为“复制旧文件 + 猜约定”。

### 6.2 模板的核心目标

模板不是为了自动生成代码，而是为了：

- 强制展示架构切面；
- 固定参数组织方式；
- 固定冷热路径边界；
- 固定错误出站方式；
- 固定状态更新方式；
- 固定文档与代码的对齐关系。

### 6.3 建议建立双模板体系

#### A. 文档模板
用于设计说明阶段：
- 域级合同卡模板
- 接口契约模板
- 三层协同表模板
- 数据流卡模板
- 正交兼容矩阵模板
- 用户子程序映射模板

#### B. 代码模板
用于实现阶段：
- Material Driver 模板
- Element Kernel 模板
- Section Evaluator 模板
- LoadBC / Constraint / Contact 模板
- Populate 模板
- Bridge 模板
- WriteBack 模板
- AI 插槽模板

### 6.4 模板子程序应显式展示什么

建议所有模板子程序按固定分区组织：

1. 责任说明区
2. 输入输出说明区
3. 前置条件区
4. 参数解包区
5. 校验区
6. 冷路径准备区
7. 热路径主计算区
8. 状态更新区
9. 错误出站区
10. 返回区

目标是让开发者从骨架层就明确：
- 什么进
- 什么出
- 什么能改
- 什么不能改
- 哪些字段必须通过 `Arg/Desc/State/Algo/Ctx` 传递

### 6.5 与 ABAQUS 用户子程序接口对齐的价值

ABAQUS 的用户子程序接口（如 UMAT / VUMAT / UEL / DLOAD / CREEP / FRIC 等）最大的价值之一，不只是“可扩展”，更重要的是：

> **接口形式反过来约束内部数据模型和成员命名。**

这对 UFC 极有借鉴意义。建议采用 **“用户子程序反推设计法”**：

1. 先明确 UFC 希望对外提供哪些用户扩展面；
2. 再根据这些扩展面反推：
   - 内部 TYPE 必须如何组织；
   - 哪些字段必须稳定命名；
   - 哪些状态必须可持久化；
   - 哪些变量应在 step / increment / iteration 粒度存在。

### 6.6 结论

模板体系与用户子程序接口设计应同步推进，不能等代码堆大后再回头补。  
对于 UFC 来说，模板与接口不是锦上添花，而是未来扩库稳定性的根基。

## 7. 正交设计：避免 Element × Section × Material × Procedure 组合爆炸

### 7.1 问题本质

如果按“每种材料 × 每种单元 × 每种截面 × 每种分析过程”写专用实现，会产生难以维护的组合爆炸。

例如表面上看可能是：

- 54 种材料
- 245 种单元
- 多类截面
- 28 / 33 / 51 类过程或分析类型

这不应被理解为要维护数万种显式实现。

### 7.2 正确思路：管理“族 + 标签 + 兼容矩阵”，而不是“笛卡尔积实现”

应将复杂度从：

`Element × Section × Material × Procedure`

改为管理：

- Element Families
- Section Families
- Material Families
- Procedure Families
- Capability Tags
- Compatibility Matrix
- Runtime Binding / Assembler

### 7.3 建议的四维正交轴

#### 轴 1：Element
负责：
- 拓扑
- 节点布局
- DOF 模式
- 插值/形函数
- B 矩阵与积分框架

不负责：
- 材料本构细节
- 截面几何定义
- 全局装配策略

#### 轴 2：Section
负责：
- 厚度 / 面积 / 惯性矩
- 层合
- 截面点组织
- 材料布置
- 方向定义

不负责：
- 单元拓扑
- 本构积分
- 全局组装

#### 轴 3：Material
负责：
- 本构关系
- 状态演化
- 切线返回
- 损伤 / 塑性 / 蠕变 / 粘弹 / 热耦合等

不负责：
- 单元拓扑
- 截面几何
- 全局求解

#### 轴 4：Procedure / Analysis Type
负责：
- 步/增量时序
- 求解流程控制
- 收敛/时间推进
- 显隐式差异
- 多物理耦合流程调度

不负责：
- 定义单元几何或材料本构内容

### 7.4 能力标签与兼容矩阵

建议所有 Element / Section / Material / Procedure 不直接用“专用乘积实现”耦合，而是通过：

1. **能力标签（Capability Tags）**
2. **兼容矩阵（Compatibility Matrix）**

进行绑定。

#### 示例：Element 需求标签
- `small_strain`
- `finite_strain`
- `3d_solid`
- `beam_like`
- `shell_layered`
- `material_driver_required`

#### 示例：Material 供给标签
- `linear_elastic`
- `elastoplastic`
- `creep`
- `damage`
- `thermal_coupled`
- `finite_strain_supported`

#### 示例：Section 提供标签
- `solid_section`
- `shell_section`
- `beam_section`
- `layered`
- `orientation_support`

#### 示例：Procedure 要求标签
- `implicit_supported`
- `explicit_supported`
- `thermal_enabled`
- `contact_nonlinear`
- `multiphysics_ready`

最终由 Binding / Assembler 层判断：
- element 需求是否被 section/material/procedure 满足；
- 某组合是否被允许。

### 7.5 截面必须提升为独立轴

单元与材料往往不是直接绑定，而是通过截面组织关联。  
因此 Section 不应继续被视为附属字段，而应作为 UFC 正交设计中的独立主轴。

这对：
- beam
- shell
- layered section
- composite section
- rebar / embedded
- orientation / thickness / integration point layout

尤其关键。

### 7.6 结论

UFC 未来要管理的，不是数万种显式组合实现，而是：

- 有限数量的族
- 一致的能力标签
- 一张清晰的兼容矩阵
- 一个稳定的运行时装配器

这才是避免组合爆炸的根本方案。

## 8. 除上述五点外，还应补充关注的关键架构问题

在“域级合同卡 / 接口契约 / 三层协同 / 模板 / 正交设计”之外，建议 UFC 还补充思考以下 10 个问题。

### 8.1 生命周期与所有权模型
必须进一步明确：
- 谁创建
- 谁持有
- 谁更新
- 谁提交
- 谁回滚
- 谁释放

特别适用于：
- State
- Ctx
- scratch buffer
- runtime workspace
- history data

否则会出现重复写、悬挂引用、冷热态混写。

### 8.2 版本演化与兼容策略
中层合同建立后，接口/字段/模板一定会演化。  
因此需要明确：
- 如何新增字段
- 如何弃用旧字段
- 如何维护向后兼容
- 如何给 guardian / harness 绑定“版本语义”

### 8.3 数据字典与统一命名
你已经敏感地注意到 UMAT/UEL 类接口会反推成员命名。  
因此 UFC 需要更明确的：
- 公共变量名词典
- 共享字段语义表
- 单位/维度规范
- state variable 命名表

否则各域合同卡会写得很像，但字段名仍可能分裂。

### 8.4 热路径与冷路径隔离制度
目前热路径要求已在讨论中涉及，但建议升格成独立制度件：
- 哪些动作永远不能出现在热路径
- 哪些字段必须提前准备
- 哪些缓存允许跨调用复用
- 哪些日志只能在冷路径采样

### 8.5 错误分级与诊断锚点
不仅要有错误码，还要有：
- 错误层级（P0/P1/P2 或 fatal/error/warn）
- 错误锚点（layer / domain / subroutine / object id）
- 错误上下文快照
- 可恢复/不可恢复分类

否则后续多域扩展后，错误会很难定位。

### 8.6 可测试性设计
建议后续每个域级合同卡都新增：
- 最小正路径测试
- 最小负路径测试
- 热路径回归点
- guardian 静态规则映射
- harness profile 绑定

让“文档不是说明书，而是测试设计输入”。

### 8.7 可观测性与调试投影
大型求解架构不能只依赖最终结果。  
需要明确：
- 域级诊断输出面
- 中间状态投影
- 局部 trace 点
- 关键 object dump 方式
- 热路径调试采样策略

### 8.8 并行 / 线程安全 / 重入性
若未来考虑并行化、任务分块、GPU 或异步调度，则每个接口契约都应逐渐明确：
- 是否可重入
- 是否线程安全
- 是否依赖共享可变状态
- 是否允许 scratch 复用

### 8.9 性能模型与缓存层级
正交设计与模板体系补起来后，还需明确：
- 哪些数据是常驻缓存
- 哪些数据按 step/inc/iter 生命周期分层
- 哪些预计算应挂在 Desc
- 哪些运行缓存应挂在 Ctx
- 哪些状态只能挂在 State

### 8.10 文档—守卫—测试三向追溯闭环
建议所有中层架构件最终都映射到：
- 文档锚点
- guardian 规则
- harness 测试入口

即形成：

**设计说明 → 规则约束 → 最小验证 → 回写文档状态**

这会让 UFC 的治理不只是“写文档”，而是真正具备可执行性。

## 11. 最终结论

本轮讨论的核心结论可以归纳为：

1. **UFC 应新增“域级合同卡”制度，作为总纲与实现之间的中层设计单元；**
2. **接口契约应与域级合同卡配套，重点约束交互，而不是替代域说明；**
3. **L3 / L4 / L5 的共享域必须同步设计，但必须坚持同域异责、合同一致、数据单向；**
4. **模板子程序 / 模板 TYPE / 用户子程序接口设计必须体系化，否则扩库必然失控；**
5. **Element / Section / Material / Procedure 应采用正交设计，通过能力标签与兼容矩阵避免组合爆炸；**
6. **除了上述五点，还必须同步考虑生命周期、版本演化、数据字典、热路径制度、错误治理、可测试性、可观测性、并行安全、性能缓存与三向追溯闭环。**

一句话总结：

> UFC 下一阶段最缺的，不是更多抽象原则，而是把“共享域怎么定义、怎么交互、怎么扩展、怎么组合、怎么验证”固化成可复用的中层架构件。

本文件即作为这套中层架构件体系的第一份沉淀说明。
