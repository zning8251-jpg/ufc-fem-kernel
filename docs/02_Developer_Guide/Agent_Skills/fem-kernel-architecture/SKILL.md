# 有限元内核架构设计技能

本技能用于在**层级-域级-功能集**、**Desc/Algo/Ctx/State 四型**、**六层架构**与**四链（理论链/逻辑链/计算链/数据链）**约束下，系统化地迭代有限元计算内核的架构设计、域级实现与验证。可与项目内既有规范（如域级深度建模、TL/UL 单元族范式）配合使用。

---

## 何时使用本技能

在以下任一情况触发本技能：

- 用户明确提到 **UFC**、**有限元内核**、**FEM kernel**、**F90 计算内核**
- 讨论 **六层架构**（L1_IF～L6_AP）、**域级**划分、**功能集**设计
- 涉及 **Desc / Algo / Ctx / State** 四型结构体封装与接口设计
- 需要打通或检查 **理论链、逻辑链、计算链、数据链**
- 进行 **架构设计**、**域级深度建模**、**算法流程开发**、**内核细节填充**
- 需要 **测试与验收**、验证体系、完备性检查
- 用户希望 **快速更新/迭代/完善** 当前架构文档与实现

## 四链视角（理论 / 逻辑 / 计算 / 数据）

架构迭代与评审时，用「四链」作为一致性检查视角：

1. **理论链**

   - 公式、本构、单元理论、时间积分理论等是否在文档与类型定义中可追溯；是否与 `UFC/docs/六层架构拆分/01-理论基础` 及同目录下理论手册一致。
2. **逻辑链**

   - 业务流程与算法流程是否清晰：Step → 载荷/BC → 单元计算 → 组装 → 求解 → 输出；各层职责是否不反转、不越界。
3. **计算链**

   - 从 L4_PH 单元计算到 L2_NM 求解器再到 L5_RT 控制的调用链是否显式、可测试；Bridge 是否集中跨层访问；是否有遗漏的接口或隐式依赖。
4. **数据链**

   - 数据从 L3_MD 定义经 Bridge/Context 进入 L4/L5/L2 的路径是否明确；Desc/Algo/Ctx/State 在每层、每域的传递是否仅通过类型名（接口只传结构体）；是否有裸数组或隐式全局状态破坏数据链可追溯性。

在**问题定位**与**设计决策**阶段，显式过一遍四链，并在文档中简短标注每条链的现状与待办。

## 域级迭代四阶段（与 99 文档对齐）

对**单个域**或**跨域的一小块**做架构迭代时，按以下四阶段执行（与 `99-域级深度建模Skill规范(补充).md` 一致）：

### 阶段 1：问题定位（Discovery）

- 画出该域（或相关域）的**全局结构草图**：顶层模型、管理器、具体对象 3～4 层，标注包含/聚合关系。
- 列出**核心概念**：核心 TYPE、关键字段、Type 间关联（1:N、M:N）。
- **对照现有实现**：是否存在对应 Type、初始化/校验逻辑；列出与规范一致点与偏差。
- **跨层/跨域依赖**：本域依赖谁（下游依赖）、谁依赖本域（上游依赖）；依赖方向是否符合六层（不反转）。
- **四链快检**：该域在理论/逻辑/计算/数据链上的位置与缺口。

**输出**：问题列表（带优先级）、现状 vs 目标对比表、依赖关系草图。

### 阶段 2：设计决策（Design）

- **核心接口**：该域对外主要查询/管理接口（Mgr 层）；签名含输入、输出、可选错误码；接口以**传结构体名**为主（Desc/Algo/Ctx/State）。
- **职责分工**：Core（数据结构与初始化/校验）、Mgr（集合与查询）、API（高层业务）、Brg（跨层桥接）的划分。
- **实现策略**：短期（补接口、不破坏现有流程）、中期（新数据组织、逐步演进）、长期（替换底层）的划分。
- **遗留处理**：硬编码、缺失 Brg、命名/接口不统一等的修复与补齐计划。
- **四链**：新接口如何支撑理论/逻辑/计算/数据链的打通。

**输出**：接口设计说明、分阶段演进路线、修复清单。

### 阶段 3：实现（Implementation）

- **优先级**：P0 修 Bug、补基础接口；P1 补齐 Mgr/Brg；P2 性能与错误处理。
- **分模块实现**：Core 补 Type/初始化；Mgr 补查询；API/Brg 按设计实现；保持**仅传结构体名**的接口约定。
- **质量**：边界检查、错误处理、复用现有实现、记录修改点与对应文档。

**输出**：修改的文件与清单、关键修改说明、可选测试。

### 阶段 4：文档沉淀（Documentation）

- 在规范中增加或更新该域的**设计纵深**：问题背景、关键决策、实现要点、接口清单、类型与过程绑定。
- **跨层依赖表**：本域依赖的上游域/接口、被哪些下游依赖；确保无循环依赖。
- 更新**域级完备性/实现状态**（如「规划中」→「已实现」→「已验证」）。
- 四链在该域的体现写入架构或数据流文档。

**输出**：更新后的规范章节、依赖表、可选依赖图。

## 四型与接口约定（速查）

- **Desc**：只读配置；接口只传类型名（如 `MD_*_Desc`）。
- **State**：读写状态；可用 `state_in` / `state_out` 等形参传递类型名。
- **Algo**：算法描述符；传类型名（如 `PH_*_Algo`）。
- **Ctx**：上下文/胶水，聚合 Desc/State/Algo 引用；接口传 `*_Ctx` 类型名。

对外子程序**形参以 TYPE 为主**，不暴露散列成员（除约定标量如 status/ierr）。成员访问封装在类型定义与类型绑定过程内部。

## 依赖与 Bridge 检查清单

- [ ] 新接口依赖的域/模块是否已列出？是否存在循环依赖（A→B→A）？
- [ ] 依赖方向是否满足六层（L1←L2←…←L6），无反向引用？
- [ ] 跨层访问是否仅通过显式 Bridge 完成？
- [ ] 是否在规范或设计文档中标注了依赖关系？

## 测试与验收（与 02-07 验证体系衔接）

- **本构/数值/物理一致性**：按 `02-07-验证体系.md` 做客观性、热力学一致性、材料对称性、Patch Test、收敛性、稳定性、能量/动量守恒等检查。
- **标准测试问题**：Cook's Membrane、Cantilever、Patch Test、弹性波等；可与 NAFEMS/ABAQUS 验证手册对照。
- **架构验收**：域级完备性检查、四链贯通检查、接口仅传结构体名检查、Bridge 无越层检查。

3. **与项目同步**：若项目新增「六层架构拆分」或 `docs/` 下的关键文档，在本技能的「项目文档锚点」表中补充路径与说明。

完成一次域级或四链相关迭代后，可简短回顾：四阶段与四链是否都过了一遍，文档与实现是否已更新。
