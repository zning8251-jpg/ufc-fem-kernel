# 技能扩展与现代化架构思考

> 在 **fem-kernel-architecture** 技能基础上，如何「增加更多技能」、增加哪些功能、如何按现代软件标准设计/开发并超越主流商业有限元软件。  
> 可与 `/bmad-brainstorming`、`/bmad-party-mode`、`/bmad-help` 等配合使用，作为迭代规划的输入。

---

## 一、主流软件的短板与“超越”的着力点

| 维度 | 主流（ABAQUS/ANSYS 等）常见问题 | 现代标准 / 可超越点 |
|------|----------------------------------|----------------------|
| **架构** | 单体、模块边界模糊、历史包袱重 | 清晰分层（你已有六层）、域级内聚、Bridge 显式跨层、可替换实现 |
| **扩展** | 脚本/插件受限、二次开发成本高 | 插件化内核、稳定 API（Desc/Ctx 即契约）、用户单元/本构注册 |
| **验证** | 验证用例分散、回归不系统 | 验证驱动开发：每算法/本构对应标准题 + 自动化回归（CI） |
| **数据与可观测** | 输入/输出格式封闭、中间状态难观测 | 开放数据契约、可追溯的数据链、诊断/性能钩子 |
| **并行与异构** | 多核/GPU 多为后补、接口不统一 | 从设计阶段考虑并行/GPU、Workspace/Backend 抽象 |
| **文档与可维护** | 文档与代码脱节、新人上手难 | 四链可追溯、规范即文档、技能包引导迭代 |
| **交付与协作** | 大版本发布、协作依赖商业流程 | 模块化发布、API 稳定性承诺、开源友好或内部清晰分界 |

**结论**：在「增加更多技能」时，优先补的是**支撑上述现代标准**的能力——验证、扩展、数据契约、可观测、并行抽象、文档与技能闭环。

## 二、建议新增技能（按功能域）

以下技能与现有 **fem-kernel-architecture** 配合使用：架构技能负责“怎么设计”，这些技能负责“在某一具体维度怎么做”。

### 2.1 验证与质量（优先）

| 技能名（建议） | 功能 | 触发词示例 | 与现代标准的关系 |
|----------------|------|-------------|------------------|
| **fem-kernel-verification** | 验证体系落地：标准题清单、收敛/稳定性/本构检查、与 NAFEMS/ABAQUS 对照、CI 回归用例设计 | 验证、回归、Patch Test、Cook's Membrane、NAFEMS、验收 | 验证驱动开发、可重复质量 |
| **fem-kernel-test-design** | 单元/集成/端到端测试设计：Mock 边界、Workspace 注入、最小可测单元 | 测试设计、单元测试、集成测试、Mock、可测性 | 可测试架构、CI 友好 |

### 2.2 数据与契约

| 技能名（建议） | 功能 | 触发词示例 | 与现代标准的关系 |
|----------------|------|-------------|------------------|
| **fem-kernel-data-contract** | 数据链契约：层间/域间输入输出类型、版本与兼容、序列化/反序列化约定 | 数据契约、接口契约、序列化、版本兼容、数据链 | 数据链可追溯、演进不破坏 |
| **fem-kernel-observability** | 可观测性：性能计数、诊断钩子、日志与追踪与 02-06 错误架构对齐 | 可观测、性能分析、诊断、日志、追踪 | 运维与调试、性能优化依据 |

### 2.3 扩展与插件

| 技能名（建议） | 功能 | 触发词示例 | 与现代标准的关系 |
|----------------|------|-------------|------------------|
| **fem-kernel-extensibility** | 扩展点设计：用户单元(UEL)、用户材料(UMAT)、用户载荷的注册与 Bridge、与 L4/L5 的对接 | UEL、UMAT、用户子程序、插件、扩展点、注册 | 超越主流的二次开发体验 |
| **fem-kernel-api-design** | 对外 API 设计：稳定接口边界、向后兼容策略、语言绑定（如 Python/C 封装）考量 | API 设计、向后兼容、Python 绑定、C 接口 | 生态与集成 |

### 2.4 并行与性能

| 技能名（建议） | 功能 | 触发词示例 | 与现代标准的关系 |
|----------------|------|-------------|------------------|
| **fem-kernel-parallel-design** | 并行与异构设计：Workspace 与线程/进程、GPU 后端抽象、与 L2_NM/L5_RT 的对接 | 并行、多线程、GPU、Workspace、后端 | 从设计阶段统一并行抽象 |

### 2.5 文档与规范

| 技能名（建议） | 功能 | 触发词示例 | 与现代标准的关系 |
|----------------|------|-------------|------------------|
| **fem-kernel-doc-traceability** | 四链与文档可追溯：理论→实现、需求→用例、变更→影响分析 | 可追溯、理论链、文档链、影响分析 | 文档即规范、可持续维护 |
| **bmad-index-docs**（已有） | 轻量索引、快速扫描 | 索引文档、快速查阅 | 与 BMad 协同 |

## 三、技能之间的依赖与使用顺序

```
fem-kernel-architecture（已有）
    ├── 域级迭代、四链、四型、Bridge/依赖
    │
    ├── fem-kernel-verification ──────── 验证体系、标准题、CI 回归
    ├── fem-kernel-test-design ───────── 可测性、Mock、测试层次
    │
    ├── fem-kernel-data-contract ────── 层间契约、数据链显式化
    ├── fem-kernel-observability ─────── 诊断、性能、错误架构
    │
    ├── fem-kernel-extensibility ─────── UEL/UMAT/扩展点、注册与 Bridge
    ├── fem-kernel-api-design ────────── 对外 API、兼容、绑定
    │
    ├── fem-kernel-parallel-design ───── Workspace、并行/GPU 抽象
    │
    └── fem-kernel-doc-traceability ──── 四链文档化、影响分析
```

- **先做**：**fem-kernel-verification**、**fem-kernel-test-design**（质量与回归是超越主流的基础）。  
- **随后**：**fem-kernel-data-contract**、**fem-kernel-observability**（数据与可观测支撑长期演进）。  
- **再补**：**fem-kernel-extensibility**、**fem-kernel-api-design**（扩展与生态）。  
- **并行可做**：**fem-kernel-parallel-design**、**fem-kernel-doc-traceability**（与实现和文档迭代同步）。

## 四、如何在此基础上完善架构、超越主流

1. **把“验证”当一等公民**  
   - 每个新算法/本构/单元族都带标准题与自动化用例；验证体系（02-07）与 CI 绑定。  
   - **fem-kernel-verification** + **fem-kernel-test-design** 支撑这一点。

2. **数据链与契约显式化**  
   - 层间、域间输入输出用类型（Desc/Ctx/State）和文档约定清楚；避免隐式全局与裸数组。  
   - **fem-kernel-data-contract** 指导契约设计与演进策略。

3. **扩展点与 Bridge 统一**  
   - 用户单元/材料/载荷通过注册与 Bridge 接入，不破坏六层边界。  
   - **fem-kernel-extensibility** 与现有 02-04 Bridge 设计对齐。

4. **可观测与错误架构一致**  
   - 性能计数、诊断钩子、错误码与 02-06 错误处理架构一致，便于运维与优化。  
   - **fem-kernel-observability** 落实这一点。

5. **并行与异构从设计阶段纳入**  
   - Workspace、Backend 抽象在 L2_NM/L5_RT 设计时就考虑，而不是事后贴补。  
   - **fem-kernel-parallel-design** 提供检查清单与模式。

6. **文档与四链同步更新**  
   - 理论链、逻辑链、计算链、数据链在规范与技能中可查；变更时做影响分析。  
   - **fem-kernel-doc-traceability** 与 **fem-kernel-architecture** 中的四链章节配合。

7. **API 稳定性与生态**  
   - 对外接口有清晰边界与兼容策略；为 Python/C 绑定留出稳定层。  
   - **fem-kernel-api-design** 指导这一点。

## 五、实施建议（如何“增加更多技能”）

1. **每次只新增 1～2 个技能**  
   - 先写 `SKILL.md`（name、description、项目文档锚点、检查清单），再在 `.claude/AGENT.md` 与 `.cursor/AGENT.md` 中登记。

2. **新技能都引用项目内已有文档**  
   - 如 02-05 数据流、02-07 验证体系、02-06 错误架构、99 域级深度建模等，技能只做“何时用、怎么用、检查什么”，不重复长文档。

3. **与 fem-kernel-architecture 联动**  
   - 在 **fem-kernel-architecture** 的「项目文档锚点」或文末增加“推荐配套技能”表，指向上述新技能，便于会话中组合使用。

4. **按优先级落地**  
   - 第一轮：**fem-kernel-verification**、**fem-kernel-test-design**（可先写 SKILL 草稿与检查清单）。  
   - 第二轮：**fem-kernel-data-contract**、**fem-kernel-observability**。  
   - 第三轮：**fem-kernel-extensibility**、**fem-kernel-api-design**、**fem-kernel-parallel-design**、**fem-kernel-doc-traceability**。

5. **定期用 BMad 复盘**  
   - 用 `/bmad-help`、`/bmad-editorial-review-structure`、`/bmad-review-adversarial-general` 对架构与技能列表做阶段评审，迭代本文档与各 SKILL.md。
