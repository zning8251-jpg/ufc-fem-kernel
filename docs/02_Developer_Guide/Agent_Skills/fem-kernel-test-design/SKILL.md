# 有限元内核测试设计技能

本技能用于在六层/域级/四型架构下，设计**可测的**单元、集成与端到端测试策略，明确 Mock 边界、注入点与最小可测单元，支撑 CI 与回归。

---

## 何时使用本技能

- 用户提到 **测试设计**、**单元测试**、**集成测试**、**端到端测试**、**可测性**
- 需要 **Mock**、**桩（stub）**、**边界注入**、**Workspace 注入**
- 讨论 **测试金字塔**、**最小可测单元**、**测试层次**
- 为**某一域或某一层**设计测试策略、或评估现有代码的可测性

## 测试层次与对应策略

| 层次 | 目标 | 策略要点 |
|------|------|----------|
| **单元** | 单模块/单子程序，无跨层 | 用最小 Desc/Ctx/State 构造输入；下层依赖用 Mock 或桩（如 L2 求解器返回给定解）；Workspace 预分配注入 |
| **集成** | 同层多域或相邻层接口 | 真实 L3→L4 或 L4→L5 数据流；Mock 仅用于更下层（如 L2 线性求解）；检查 Ctx 传递与 Bridge 调用 |
| **端到端** | 从输入到输出的完整链路 | 标准题或简化 INP；与 fem-kernel-verification 的标准题清单对齐；可重复、可自动化 |

---

## 可测性检查清单

- [ ] 接口是否**仅传结构体名**（Desc/Algo/Ctx/State）？是则易于构造最小实参。
- [ ] 是否依赖全局/模块级状态？若有，是否可改为通过 Ctx 或 Workspace 注入？
- [ ] 跨层调用是否经 Bridge？是则可在 Bridge 边界做 Mock 或桩。
- [ ] 是否有 Workspace/临时数组？是否支持从外部传入以便固定规模、重复执行？
- [ ] 错误路径是否可通过注入错误码或非法输入触发并断言？

---

## 产出建议

- **测试矩阵**：域/层 × 单元/集成/端到端，标注已覆盖、待补充、依赖 Mock。
- **Mock 清单**：每个被测对象依赖的下层接口、建议的桩行为（如线性求解返回给定 u）。
- **CI 触发**：哪些测试在提交到某路径时运行，与 fem-kernel-verification 的回归用例如何分工。

---

## 与 fem-kernel-architecture / fem-kernel-verification 的衔接

- **架构**：在「阶段 3：实现」中，每新增接口同时考虑单元/集成测试用例；依赖检查清单中的模块即 Mock 候选边界。
- **验证**：验证技能负责「测什么（标准题、正确性）」；本技能负责「怎么测（分层、隔离、注入、可自动化）」。二者结合得到完整 CI 策略。
