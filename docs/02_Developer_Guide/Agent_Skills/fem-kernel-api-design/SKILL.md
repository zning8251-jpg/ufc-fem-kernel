# 有限元内核 API 设计技能

本技能用于划定内核的**对外稳定 API** 与**内部实现**边界，制定向后兼容与废弃策略，并考虑语言绑定（如 Python/C）所需的封装层，支撑生态与长期演进。

---

## 何时使用本技能

- 用户提到 **API 设计**、**对外接口**、**稳定 API**、**内部实现边界**
- 讨论 **向后兼容**、**废弃**、**迁移**、**版本策略**
- 需要 **Python 绑定**、**C 接口**、**FFI**、**封装层** 设计
- 划定哪些模块/类型/子程序对**外部用户或上层应用**可见、哪些为内部实现

## API 设计维度

| 维度 | 内容 |
|------|------|
| **边界划分** | 哪些子程序/类型为 **公开 API**（承诺稳定）、哪些为 **内部**（可随时变更）；通常 L6_AP 入口、部分 Bridge、以及明确标注的「稳定类型」为 API |
| **向后兼容** | 新增参数使用可选或默认值；废弃先标记为 deprecated 并给出替代 API 与迁移期；避免移除或改变已有参数的语义 |
| **版本策略** | API 版本号（如主.次）与语义；何时允许破坏性变更（新主版本）、如何提供兼容层或迁移指南 |
| **语言绑定** | C 接口：导出 C 兼容的符号与类型，便于 FFI；Python：通过 C 扩展或 Cython 封装，仅暴露稳定 API 子集；错误与状态在绑定层映射为语言惯用形式 |

## API 设计检查清单

- [ ] 公开 API 是否有**文档化列表**（模块名、子程序/类型名、简要用途）？
- [ ] 是否明确**废弃流程**（标记、替代 API、迁移期、最终移除版本）？
- [ ] 新增 API 是否考虑**默认值/可选参数**，避免破坏已有调用方？
- [ ] 若提供 C/Python 绑定：**导出集合**是否与「稳定 API」一致，错误码/异常是否与 02-06 或绑定层约定一致？
- [ ] 内部模块是否**不直接暴露**给绑定层（仅通过稳定 API 间接使用）？

---

## 产出建议

- **公开 API 清单**：按层/域或按功能列出的稳定子程序与类型；标注引入版本、废弃（若有）。
- **兼容与废弃策略**：一页纸说明版本号含义、何时可破坏性变更、如何废弃与迁移。
- **绑定层约定**：C 导出函数前缀、错误返回约定；Python 模块/类与内核 API 的对应、异常与 status 的映射。

## 与 fem-kernel-architecture / fem-kernel-data-contract 的衔接

- **架构**：API 边界通常对应 L6_AP 或显式 API 层与 Bridge 门面；新增公开 API 时检查依赖方向与 Bridge 职责。
- **数据契约**：公开 API 使用的类型即契约的一部分；类型与字段的稳定/版本策略与 fem-kernel-data-contract 一致，并在契约表中标注「对外可见」。
