# 有限元内核扩展性技能

本技能用于在六层架构下设计**扩展点**：用户单元（UEL）、用户材料（UMAT）、用户载荷等通过注册与 Bridge 接入，不破坏层边界，形成稳定、可演进的二次开发接口。

---

## 何时使用本技能

- 用户提到 **UEL**、**UMAT**、**用户子程序**、**用户材料**、**用户载荷**
- 讨论 **插件**、**扩展点**、**注册**、**二次开发**、**扩展接口**
- 需要设计或评审 **L4_PH/L5_RT 与用户代码的边界**、**回调约定**、**数据格式**

## 扩展点类型与对接层

| 扩展类型 | 典型对接层/域 | 契约要点 |
|----------|----------------|----------|
| **UEL（用户单元）** | L4_PH Elem + L5_RT Asm/Elem | 单元刚度/质量/残差接口；节点数、自由度、积分规则由内核描述，用户实现计算；通过 Brg 注册与调度 |
| **UMAT（用户材料）** | L4_PH Mat + L3_MD Mat 描述 | 应力/切线刚度接口；应变/状态由内核传入，用户更新应力与 D；通过 Brg 注册与路由 |
| **用户载荷/BC** | L3_MD LoadBC + L5_RT 装配 | 载荷/BC 值接口；时间、位置、DOF 由内核提供，用户返回力或约束值；通过 Brg 注册与调用 |

## 扩展点设计检查清单

- [ ] 扩展接口是否**仅使用稳定类型**（Desc/Ctx/State 或明文的扩展用类型）传递，无内核内部类型泄漏？
- [ ] **注册与调度**是否集中在 Bridge 层（如 L4_PH 的 *_Brg、L5_RT 的 UEL_Reg），不分散在各 Core？
- [ ] 用户代码与内核的**调用约定**（参数顺序、数组布局、错误返回）是否文档化且版本可兼容？
- [ ] 是否避免用户代码**反向依赖**内核实现细节（仅依赖公开接口与契约）？
- [ ] 错误与可观测性：用户子程序内的错误是否通过**统一错误码或 Ctx** 返回，与 02-06 一致？

## 产出建议

- **扩展接口表**：每类扩展（UEL/UMAT/载荷等）的入口签名、输入/输出类型、注册 API、实现位置（Brg 模块）。
- **与 02-04 的对应**：每个扩展点对应哪种 Bridge 模式（适配器/门面/路由），在 02-04 或扩展设计文档中注明。
- **版本与兼容**：扩展接口的稳定字段、新增参数的可选/默认策略，与 fem-kernel-data-contract 衔接。

## 与 fem-kernel-architecture / fem-kernel-data-contract 的衔接

- **架构**：扩展仅通过 Bridge 接入，不破坏 L1←…←L6 依赖方向；新增扩展类型时在域级规划中补充 Brg 与契约。
- **数据契约**：扩展的输入/输出类型纳入契约表，变更时做兼容性评估与文档更新。
